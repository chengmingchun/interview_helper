Java 多线程基础：面试官想听的不是“会创建线程”

创建方式

Thread / Runnable / Callable

  • Thread：直接继承，简单但侵入性强。
  • Runnable：任务和线程分离，最常见。
  • Callable：可以返回结果、抛异常，通常配合 Future。
  • 生产里更推荐交给线程池，而不是频繁 new Thread。
线程状态

NEW 到 TERMINATED

  • NEW：创建但未 start。
  • RUNNABLE：可运行，等待 CPU 调度也算这里。
  • BLOCKED：等待 monitor 锁。
  • WAITING/TIMED_WAITING：等待唤醒或超时。
  • TERMINATED：执行结束。
核心风险

并发问题本质

  • 共享变量读写：原子性、可见性、有序性。
  • 共享资源竞争：锁粒度、死锁、饥饿。
  • 任务调度失控：线程过多、队列堆积、下游被打爆。
  • 上下文污染：ThreadLocal 未清理、异步上下文丢失。

线程池参数：七参数其实是一套资源预算

标准回答不要只背参数名，要讲任务从提交到执行的路径。

七个参数怎么记

  • corePoolSize：核心线程数，常驻处理稳定流量。
  • maximumPoolSize：最大线程数，应对队列满后的突增流量。
  • keepAliveTime + unit：非核心线程空闲多久回收。
  • workQueue：任务缓冲区，决定削峰方式和内存风险。
  • threadFactory：线程命名、优先级、异常处理，排障必备。
  • handler：拒绝策略，系统满载后的业务选择。

执行流程必须背成肌肉记忆

  1. 运行线程数小于 core，创建核心线程执行。
  2. 核心线程满了，任务进入队列。
  3. 队列满了，且线程数小于 max，创建非核心线程。
  4. 线程达到 max 且队列满，触发拒绝策略。

注意：不是任务一来就扩到最大线程。先 core，再 queue，再 max，最后 reject。

new ThreadPoolExecutor(
    corePoolSize,
    maximumPoolSize,
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(queueCapacity),
    namedThreadFactory,
    new ThreadPoolExecutor.CallerRunsPolicy()
);

拒绝策略：系统满了以后，业务到底愿意付出什么代价

四种内置策略

  • AbortPolicy：抛 RejectedExecutionException，默认策略。核心任务推荐显式失败、告警、补偿。
  • CallerRunsPolicy：调用者线程执行任务，让上游变慢，形成背压。
  • DiscardPolicy：静默丢弃。只能用于可丢任务，如部分埋点、采样日志。
  • DiscardOldestPolicy：丢掉队列最老任务，再重试提交。适合过期就无价值的任务。

怎么回答“你会怎么选”

  • 核心交易、订单、支付：不能静默丢，Abort + 告警 + 补偿。
  • 异步削峰：可用 CallerRuns 给入口背压，但要避免拖死 Web 线程。
  • 日志、埋点：可采样丢弃，但必须有丢弃指标。
  • 实时刷新类任务：可丢旧任务，保留最新任务。

一句话：拒绝策略不是线程池配置项，而是业务可靠性决策。

JMM 与 volatile：并发 bug 很多不是锁问题，是内存语义问题

原子性

一个操作不可被拆分。i++ 包含读、改、写，不是原子操作。可以用 synchronized、Lock 或 AtomicInteger。

可见性

线程 A 改了共享变量，线程 B 能及时看到。volatile、锁释放/获取都能建立可见性。

有序性

编译器和 CPU 可能重排序。volatile 通过内存屏障约束关键重排序，DCL 单例必须用它。

class Singleton {
  private static volatile Singleton instance;

  static Singleton getInstance() {
    if (instance == null) {
      synchronized (Singleton.class) {
        if (instance == null) {
          instance = new Singleton();
        }
      }
    }
    return instance;
  }
}

锁机制：别只会说 synchronized 和 ReentrantLock 的区别

synchronized

  • 基于对象监视器 monitor，进入临界区加锁，退出自动释放。
  • 可重入，异常时自动释放锁。
  • 锁对象可以是实例对象、Class 对象或任意对象。
  • JVM 有偏向锁、轻量级锁、重量级锁等优化，不要简单说它慢。

常见追问

问：锁升级是什么？

答：竞争从低到高时，锁会经历无锁、偏向锁、轻量级锁、重量级锁，目标是在低竞争下减少系统调用和阻塞成本。

ReentrantLock

  • 基于 AQS，可重入，支持公平锁和非公平锁。
  • 支持 tryLock、超时获取、可中断获取锁。
  • 支持多个 Condition 条件队列。
  • 必须在 finally 里 unlock。

怎么选

简单同步优先 synchronized；需要超时、可中断、公平锁、多条件队列时用 ReentrantLock。

死锁四条件

  • 互斥：资源一次只能被一个线程持有。
  • 占有且等待：拿着一个资源等待另一个资源。
  • 不可抢占：资源不能被强制夺走。
  • 循环等待：线程之间形成等待环。

修复思路

  • 固定加锁顺序。
  • 缩小锁粒度和锁持有时间。
  • 使用 tryLock 超时退出。
  • 通过 jstack 查 BLOCKED 和 deadlock 信息。

JUC 工具：把工具和业务场景绑定，才好记

AQS

state + CAS + 队列

AQS 用 volatile state 表示同步状态，用 CAS 修改状态，获取失败的线程进入 FIFO 等待队列。

协调工具

Latch / Barrier / Semaphore

  • CountDownLatch：等待多个任务完成，不能复用。
  • CyclicBarrier：一组线程相互等待，可复用。
  • Semaphore：限制并发访问量，常用于资源隔离。
上下文

ThreadLocal

适合 traceId、用户上下文、租户信息。在线程池里必须 finally remove，否则会上下文污染或内存泄漏。

try {
  USER_CONTEXT.set(user);
  doBusiness();
} finally {
  USER_CONTEXT.remove();
}
