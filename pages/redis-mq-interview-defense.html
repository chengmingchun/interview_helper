Redis / MQ 高频防线

点击展开追问，每次只练一个主题。

把你的订单项目接到 MQ 答案里

你不是没有经验，而是没有把经验翻译成标准问题。

1. 接收订单入口校验、业务唯一键、权限与参数校验。
2. 本地落库订单状态初始化，数据库事务保证核心数据先存在。
3. 可靠发消息事务消息或 Outbox，避免“库成功、消息没发”。
4. Broker 持久化确认机制、副本与持久化降低消息丢失概率。
5. 幂等消费业务键、唯一索引、状态机防止重复下发。
6. 重试 / 补偿短暂失败重试；长期失败转补偿或人工。
7. 链路闭环消息 ID、订单 ID、状态变化、失败原因可追踪。
项目答法我把可靠性拆成发送端、Broker 和消费端三层。发送端通过事务消息或本地消息表避免数据库提交后消息丢失；Broker 开启持久化和副本；消费端采用手动确认、失败重试和死信队列。由于消息系统通常只能做到至少一次投递，业务端必须依靠订单唯一键、数据库唯一索引和状态机保证幂等，最后再用定时补偿和链路日志兜底。

五类场景题，固定从链路排查

先定位在哪一层出问题，再给配置和补偿动作。

消息丢失

生产 → Broker → 消费

生产确认与重试；Broker 持久化和副本；处理成功后确认；失败进入重试、死信和补偿。

重复消费

接受至少一次，业务做幂等

业务唯一键、数据库唯一索引、状态机和去重表；Redis 去重只能辅助。

消息积压

先看生产速率和消费速率

检查 lag、失败率、消费耗时和下游 DB/接口，临时扩容、降级，随后修根因。

缓存不一致

数据库是权威源

先更新数据库再删缓存；删除失败重试或订阅 binlog；TTL 兜底，强一致读绕过缓存。

分布式锁异常

锁不自动保证业务正确

唯一 value、防误删、Lua 原子删除、续期或合理 TTL，最终用唯一约束与状态机兜底。

热点 Key

先区分读热点和写热点

读热点可本地缓存与请求合并；写热点需拆分、限流或串行化，但不能破坏原子性。
