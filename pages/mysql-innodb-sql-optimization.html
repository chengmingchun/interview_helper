1. 介绍项目中的数据库设计与 SQL 优化经验

这个问题不是让你背 MySQL，而是让你证明：你能从业务模型、数据规模、查询路径和索引设计角度思考。

可直接背： 我之前项目里比较典型的是订单/云资源编排相关表设计。数据库设计上，我会先从业务实体和状态流转出发，拆出订单主表、订单明细表、资源实例表、租户/用户维度表、操作日志表。订单主表保存订单状态、租户、用户、创建时间、业务唯一号等主流程字段；明细表保存资源规格、实例 ID、下发参数；日志表记录状态变更和失败原因，避免主表被大量非核心字段污染。 索引设计上，我不会只给每个字段单独建索引，而是围绕核心查询场景建联合索引。比如按照租户 + 状态 + 创建时间查询订单列表，可以设计 tenant_id, status, create_time；按照业务单号查幂等结果，可以用 request_no 唯一索引；按照订单 ID 查明细，可以让 order_id 成为明细表索引。这样索引是服务查询路径的，而不是服务字段存在感的。 SQL 优化上，我通常先看慢 SQL 和接口耗时，再看 Explain 里的 type、key、rows、filtered、Extra，判断有没有全表扫描、索引失效、回表过多、排序临时表、分页过深等问题。我实际遇到过复杂 SQL 超时问题，通过调整索引、改写查询、减少回表和优化分页方式，把几十秒级查询优化到秒级以内。

2. InnoDB 聚簇索引、二级索引底层结构，回表查询完整流程

这题必须说清：InnoDB 表数据本身就存放在聚簇索引的叶子节点上，二级索引叶子节点存的是主键值。

聚簇索引

按照主键组织的 B+ 树。叶子节点保存完整行数据。一张 InnoDB 表只能有一个聚簇索引，因为数据只能按一种方式物理组织。

Primary Key

二级索引

非主键索引也是 B+ 树。叶子节点保存索引列值 + 主键值，不保存完整行。通过二级索引查到主键后，再去聚簇索引查完整行，这就是回表。

Secondary Index

回表完整流程

select id, name, age, address from user where name = 'Tom';

1. 走 idx_name 二级索引 B+ 树
2. 在 idx_name 叶子节点找到 name = Tom 的记录
3. 叶子节点中拿到主键 id
4. 根据 id 回到聚簇索引 B+ 树
5. 在聚簇索引叶子节点拿到完整行数据
6. 返回 id, name, age, address

为什么 InnoDB 推荐自增主键

  • 聚簇索引按主键有序组织，自增主键插入通常追加到末尾，页分裂少。
  • 随机 UUID 会导致插入位置随机，容易页分裂、页抖动、缓存命中变差。
  • 二级索引叶子节点保存主键值，主键过大会让所有二级索引变大。
60 秒回答：InnoDB 的主键索引是聚簇索引，叶子节点保存完整行数据；二级索引的叶子节点保存索引列和主键值。如果查询条件命中二级索引，但 select 字段不在该二级索引里，就需要先通过二级索引找到主键，再根据主键回到聚簇索引查整行，这就是回表。覆盖索引能避免回表。

3. MVCC 实现原理：undo log 版本链 + ReadView 如何实现隔离？

MVCC 解决的是读写并发问题：读不阻塞写，写不阻塞普通快照读。

核心组件

组件作用
trx_id每行记录隐藏字段，表示最后修改该行的事务 ID。
roll_pointer隐藏字段，指向 undo log 中的历史版本。
undo log保存旧版本数据，用于回滚和构造历史版本。
ReadView当前快照读能看到哪些事务版本的判断规则。

版本链

当前记录: value = C, trx_id = 30, roll_pointer -> undo_2
undo_2: value = B, trx_id = 20, roll_pointer -> undo_1
undo_1: value = A, trx_id = 10, roll_pointer -> null

查询时不是简单读最新行，而是结合 ReadView 判断当前版本能不能看；看不到就沿着 undo log 版本链往前找，直到找到可见版本。

ReadView 关键字段

字段含义
m_ids生成 ReadView 时活跃事务 ID 列表。
min_trx_id活跃事务中的最小事务 ID。
max_trx_id下一个将要分配的事务 ID。
creator_trx_id创建这个 ReadView 的事务 ID。

可见性判断简化版

if row.trx_id == creator_trx_id:
    visible  # 自己改的自己能看见
else if row.trx_id < min_trx_id:
    visible  # 在快照前已经提交
else if row.trx_id >= max_trx_id:
    invisible # 快照后才出现
else if row.trx_id in m_ids:
    invisible # 快照生成时还没提交
else:
    visible

RC 和 RR 的区别

隔离级别ReadView 生成时机现象
Read Committed每次快照读都生成新的 ReadView。同一事务内两次读可能读到不同已提交数据。
Repeatable Read事务第一次快照读生成 ReadView，后续复用。同一事务内多次快照读结果一致。
60 秒回答：MVCC 依赖行记录隐藏字段、undo log 版本链和 ReadView。每次更新会生成旧版本到 undo log，当前行通过 roll_pointer 指向历史版本。快照读时，InnoDB 根据 ReadView 判断当前版本是否可见；如果不可见，就沿 undo log 版本链找上一个版本。RC 下每次查询生成新的 ReadView，RR 下事务第一次快照读生成 ReadView 并复用，所以 RR 能实现可重复读。

4. MySQL RR 隔离级别如何解决幻读？间隙锁、临键锁触发场景

先区分：普通快照读靠 MVCC 避免幻读；当前读靠 next-key lock 避免幻读。

快照读

普通 select，不加锁。RR 下复用同一个 ReadView，所以同一事务内多次读结果一致。

MVCC

当前读

select ... for update、update、delete、insert 需要读最新数据，会加锁。RR 下依赖记录锁、间隙锁、临键锁阻止其他事务插入幻影记录。

Lock

三种锁

锁锁什么作用
Record Lock锁索引记录本身。防止已有记录被修改。
Gap Lock锁两个索引记录之间的间隙。防止间隙内插入新记录。
Next-Key LockRecord Lock + Gap Lock。锁左开右闭区间，防止当前读幻读。

触发例子

-- 表中 age 索引已有值：10, 20, 30
-- 事务 A
select * from user where age between 10 and 20 for update;

-- 可能锁住：
-- 记录 10、20
-- 以及相关间隙，阻止其他事务插入 age=15 的新记录

为什么必须走索引

InnoDB 的行锁是加在索引上的。如果条件没有走索引，可能退化成大量记录加锁甚至类似锁表效果，影响并发。

60 秒回答：RR 下解决幻读要分情况。普通 select 是快照读，依靠 MVCC 和事务级 ReadView，让同一事务内多次查询看到一致快照。select for update、update、delete 属于当前读，要读最新数据，所以不能只靠 MVCC，需要通过 next-key lock，也就是记录锁加间隙锁，锁住命中的索引记录和记录之间的范围，阻止其他事务插入满足条件的新行，从而避免幻读。

5. 死锁产生条件，如何定位和避免数据库死锁？

死锁四个条件

条件说明
互斥资源一次只能被一个事务持有。
占有且等待事务持有一部分锁，同时等待其他锁。
不可抢占锁不能被强行剥夺，只能事务释放。
循环等待T1 等 T2，T2 又等 T1。

经典死锁

事务 A:
update account set balance = balance - 100 where id = 1;
update account set balance = balance + 100 where id = 2;

事务 B:
update account set balance = balance - 50 where id = 2;
update account set balance = balance + 50 where id = 1;

A 先锁 id=1 再等 id=2；B 先锁 id=2 再等 id=1，形成循环等待。

如何定位

show engine innodb status\G;

-- 关注：
LATEST DETECTED DEADLOCK
TRANSACTION
WAITING FOR THIS LOCK TO BE GRANTED
HOLDS THE LOCK(S)
SQL statement

线上还要结合慢 SQL 日志、应用日志 traceId、事务开始和结束时间、接口调用链路，判断谁持有锁太久。

如何避免

  • 固定加锁顺序，例如转账永远先锁小 id，再锁大 id。
  • 事务尽量短，不要在事务里做远程调用、慢查询、大批量循环。
  • where 条件必须走索引，避免锁范围扩大。
  • 批量更新拆小批，减少锁持有时间。
  • 合理设置锁等待超时，捕获死锁异常后重试。
  • 用唯一索引精准命中，减少 gap lock 范围。

6. 联合索引最左匹配原则，详细列举索引失效场景

联合索引不是多个单列索引的简单叠加。它按照从左到右的字段顺序构建 B+ 树。

index idx_user_status_time(user_id, status, create_time)
SQL能否充分使用索引原因
where user_id = ?能命中最左字段。
where user_id = ? and status = ?能连续命中左前缀。
where user_id = ? and status = ? order by create_time通常能过滤和排序都符合索引顺序。
where status = ?不能充分使用跳过最左字段 user_id。
where user_id = ? and create_time > ?只能部分使用跳过 status，create_time 利用受限。
where user_id > ? and status = ?status 难以继续用于定位范围查询后面的字段通常无法继续用于索引定位。

索引失效高频场景

场景例子原因/处理
左模糊like '%abc'无法从 B+ 树有序前缀定位；可用倒排索引/ES/改成右模糊。
函数包裹字段date(create_time) = '2026-06-29'对索引列计算导致无法直接使用；改成范围查询。
隐式类型转换varchar phone = 13800138000字段被转换，可能导致索引失效；类型保持一致。
对字段运算amount + 1 = 100索引列参与表达式计算；改写为 amount = 99。
OR 条件a = 1 or b = 2如果两边不是都有合适索引，可能全表扫；可改 union。
IN 太大id in (...大量值...)优化器可能判断成本过高；分批或临时表 join。
不等于status != 1选择性差，可能不走索引。
排序字段不符合索引顺序where a=? order by c,b和联合索引顺序不一致，可能 filesort。
60 秒回答：联合索引遵循最左匹配，因为 B+ 树是按联合字段从左到右排序的。查询必须从最左字段开始连续匹配，遇到范围查询后，后面的字段通常不能继续用于索引定位。常见失效包括左模糊、函数包裹、隐式类型转换、字段计算、跳过最左列、or 两边索引不完整、in 过大、低选择性字段、不符合索引顺序的排序。

7. 覆盖索引原理，为什么能极大提升查询性能？

覆盖索引的核心收益：查询所需字段都在二级索引叶子节点里，不需要回表。

index idx_name_age(name, age)

select name, age from user where name = 'Tom';
-- name 和 age 都在 idx_name_age 中，直接返回，不回表

select name, age, address from user where name = 'Tom';
-- address 不在索引里，需要回表

为什么快

  • 减少一次聚簇索引查询，降低随机 IO。
  • 二级索引通常比整行数据小，缓存命中更高。
  • 避免读取大字段，例如 text、json、varchar 大字段。
  • 分页场景可以先用覆盖索引筛出主键，再回表少量数据。

深分页优化例子

-- 差：offset 很大时要扫描并丢弃大量行，还可能大量回表
select * from order_info
where tenant_id = 1
order by create_time desc
limit 100000, 20;

-- 优化：先走覆盖索引拿 id，再回表 20 行
select o.*
from order_info o
join (
  select id from order_info
  where tenant_id = 1
  order by create_time desc
  limit 100000, 20
) t on o.id = t.id;
面试表达：覆盖索引不是一种新的索引类型，而是一种查询效果。只要查询字段都能从某个索引中拿到，就可以避免回表。它对高频列表查询、分页查询、统计查询非常有效，但不能为了覆盖所有字段把索引建得过宽，否则会增加写入成本和索引维护成本。

9. 千万级数据表优化思路，是否需要分库分表？字段如何取舍？

千万级不等于一定分库分表。先判断瓶颈：查询慢、写入慢、索引太大、单表容量、热点、归档、还是业务增长不可控。

优化优先级

  1. 先确认 SQL 和索引是否合理：避免全表扫、回表过多、排序临时表。
  2. 做字段裁剪：列表页不要 select *，大字段拆出去。
  3. 冷热分离：历史数据归档，核心表只保留热数据。
  4. 读写分离：读多写少可以引入从库，但要处理主从延迟。
  5. 缓存：高频读用 Redis/本地缓存，但要处理一致性和击穿。
  6. 分区表：按时间等维度管理历史数据，但不能解决所有查询问题。
  7. 分库分表：当单表/单库容量、写入、索引、运维都成为瓶颈时再做。

字段如何取舍

原则说明
核心字段放主表状态、租户、用户、时间、金额、主流程字段。
大字段拆表text、json、长 varchar、请求响应报文放扩展表。
低频字段拆表详情页才看的字段不要拖慢列表页。
冗余要克制读性能可以用冗余换，但要有一致性维护策略。
索引字段短小稳定过长字段会放大索引体积，频繁变化字段维护成本高。

什么时候分库分表

适合不适合
数据量持续增长，历史归档后仍然巨大。只是某条 SQL 没索引。
单库写入压力明显成为瓶颈。数据量大但查询都能命中索引。
业务天然有分片键，如 user_id、tenant_id。大量跨分片 join、分页、聚合。
可以接受分布式 ID、路由、聚合查询复杂度。团队还没有分库分表运维经验。
面试金句：分库分表是最后手段，不是第一反应。因为它会引入分布式 ID、分片路由、跨分片查询、跨分片事务、扩容迁移和运维复杂度。千万级表先看 SQL、索引、字段宽度、冷热数据、缓存、归档和读写分离。只有当单表容量和单库写入真的成为主要瓶颈，并且业务有稳定分片键时，再考虑分库分表。

10. 算法手撕：中等 - 最长无重复子串

这是滑动窗口经典题。核心是右指针扩张，左指针跳到重复字符上次出现位置的下一位。

Java 写法

public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> last = new HashMap<>();
    int left = 0;
    int ans = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (last.containsKey(c) && last.get(c) >= left) {
            left = last.get(c) + 1;
        }
        last.put(c, right);
        ans = Math.max(ans, right - left + 1);
    }

    return ans;
}

数组优化写法

public int lengthOfLongestSubstring(String s) {
    int[] last = new int[128];
    Arrays.fill(last, -1);
    int left = 0, ans = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (last[c] >= left) {
            left = last[c] + 1;
        }
        last[c] = right;
        ans = Math.max(ans, right - left + 1);
    }
    return ans;
}

Rust 写法

use std::collections::HashMap;

pub fn length_of_longest_substring(s: String) -> i32 {
    let chars: Vec<char> = s.chars().collect();
    let mut last: HashMap<char, usize> = HashMap::new();
    let mut left = 0usize;
    let mut ans = 0usize;

    for right in 0..chars.len() {
        let c = chars[right];
        if let Some(&idx) = last.get(&c) {
            if idx >= left {
                left = idx + 1;
            }
        }
        last.insert(c, right);
        ans = ans.max(right - left + 1);
    }

    ans as i32
}

为什么 left 不能回退

s = abba
right = 2, c = b, left 从 0 跳到 2
right = 3, c = a, a 上次在 0，但 0 已经不在窗口内
所以 left 不能回到 1，必须取 max 或判断 last[c] >= left
复杂度：每个字符最多进窗口一次、出窗口一次，时间 O(n)，空间 O(字符集大小)。面试解释时重点说清窗口含义：窗口 [left, right] 内始终没有重复字符。

最后：10 题口述模板

1. 项目数据库设计：从业务实体、状态流转、主表明细表、索引设计、幂等唯一键、慢 SQL 排查结果讲。
2. InnoDB 索引：聚簇索引叶子节点存整行，二级索引叶子节点存索引列和主键。二级索引查非覆盖字段要回表。
3. MVCC：隐藏字段 trx_id 和 roll_pointer 指向 undo log 版本链，ReadView 判断版本可见性，RC 每次生成，RR 事务级复用。
4. RR 幻读：快照读靠 MVCC，当前读靠 next-key lock，也就是记录锁加间隙锁，阻止范围内插入新记录。
5. 死锁：互斥、占有等待、不可抢占、循环等待。定位看 show engine innodb status，避免靠固定加锁顺序、短事务、索引命中。
6. 联合索引：按字段顺序构建 B+ 树，必须从最左字段连续匹配。左模糊、函数、隐式转换、or、范围后字段都要警惕。
7. 覆盖索引：查询字段都在索引里即可避免回表，减少随机 IO，列表页和深分页优化很常用。
8. Explain：重点看 type、key、key_len、rows、filtered、Extra，关注 ALL、Using temporary、Using filesort。
9. 千万级优化：先 SQL/索引/字段/冷热/缓存/归档，再考虑分库分表。分库分表会引入路由、事务、聚合、扩容复杂度。
10. 最长无重复子串：滑动窗口。重复字符在窗口内时，left 跳到上次位置 + 1；每轮更新 last 和答案。
