2. IoC / DI：容器帮你管理对象关系

IoC 是控制反转：对象不再自己 new 依赖，而是把创建、装配、生命周期交给容器。DI 是 IoC 最常见的实现方式：通过构造器、setter 或字段把依赖注入进来。

2.1 不推荐：业务类自己 new 依赖

public class OrderService {
    private final InventoryClient inventoryClient = new InventoryClient();
    private final PaymentClient paymentClient = new PaymentClient();

    public void createOrder(OrderCommand command) {
        inventoryClient.lock(command.skuId(), command.count());
        paymentClient.prePay(command.userId(), command.amount());
    }
}

问题：强耦合、难替换、难测试、难加代理增强，也很难按环境切换实现。

2.2 推荐：构造器注入

@Service
public class OrderService {
    private final InventoryClient inventoryClient;
    private final PaymentClient paymentClient;

    public OrderService(InventoryClient inventoryClient, PaymentClient paymentClient) {
        this.inventoryClient = inventoryClient;
        this.paymentClient = paymentClient;
    }

    public void createOrder(OrderCommand command) {
        inventoryClient.lock(command.skuId(), command.count());
        paymentClient.prePay(command.userId(), command.amount());
    }
}

2.3 多实现注入：按名称或 @Qualifier

public interface SmsSender {
    void send(String phone, String text);
}

@Component("aliyunSmsSender")
class AliyunSmsSender implements SmsSender {
    public void send(String phone, String text) {}
}

@Component("tencentSmsSender")
class TencentSmsSender implements SmsSender {
    public void send(String phone, String text) {}
}

@Service
class LoginService {
    private final SmsSender smsSender;

    LoginService(@Qualifier("aliyunSmsSender") SmsSender smsSender) {
        this.smsSender = smsSender;
    }
}

2.4 配置属性绑定

@ConfigurationProperties(prefix = "pay")
public record PayProperties(
    String appId,
    String secret,
    Duration timeout
) {}

@Configuration
@EnableConfigurationProperties(PayProperties.class)
class PayConfig {
}
构造器注入为什么推荐：依赖不可变、必需依赖一眼可见、便于单元测试，也能更早暴露循环依赖。

3. Bean 生命周期：从 BeanDefinition 到可用对象

Bean 生命周期是 Spring 面试的骨架题。你不需要背每个扩展点的源码行号，但要讲出主流程和关键扩展点。

  1. 扫描配置，生成 `BeanDefinition`。
  2. 实例化 Bean：构造器或工厂方法。
  3. 属性填充：依赖注入。
  4. 执行 Aware 回调：如 `BeanNameAware`、`ApplicationContextAware`。
  5. 执行 `BeanPostProcessor#postProcessBeforeInitialization`。
  6. 初始化：`@PostConstruct`、`InitializingBean`、`init-method`。
  7. 执行 `BeanPostProcessor#postProcessAfterInitialization`，AOP 代理常在这里生成。
  8. 容器关闭时销毁：`@PreDestroy`、`DisposableBean`、`destroy-method`。

生命周期示例

@Component
public class ReportClient implements InitializingBean, DisposableBean {
    @PostConstruct
    public void postConstruct() {
        System.out.println("1. @PostConstruct");
    }

    @Override
    public void afterPropertiesSet() {
        System.out.println("2. afterPropertiesSet");
    }

    @PreDestroy
    public void preDestroy() {
        System.out.println("3. @PreDestroy");
    }

    @Override
    public void destroy() {
        System.out.println("4. destroy");
    }
}

BeanPostProcessor：统一增强 Bean

@Component
public class ClientLogBeanPostProcessor implements BeanPostProcessor {
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (beanName.endsWith("Client")) {
            System.out.println("client bean ready: " + beanName);
        }
        return bean;
    }
}

4. AOP 与代理：横切逻辑不要散落业务代码

AOP 适合日志、监控、事务、权限、幂等、限流等横切逻辑。Spring AOP 基于代理，默认只拦截 Spring Bean 的方法调用。

4.1 自定义耗时日志切面

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface CostLog {
}

@Aspect
@Component
public class CostLogAspect {
    @Around("@annotation(CostLog)")
    public Object logCost(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return pjp.proceed();
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println(pjp.getSignature() + " cost " + cost + "ms");
        }
    }
}

4.2 使用

@Service
public class PricingService {
    @CostLog
    public BigDecimal calculatePrice(Long skuId) {
        return BigDecimal.TEN;
    }
}

4.3 AOP 失效场景

场景为什么处理
同类内部方法调用`this.method()` 没走代理对象拆到另一个 Bean，或通过代理调用
方法不是 public代理拦截能力受限把切点方法设计为 public
对象不是 Spring Bean容器没机会生成代理交给 Spring 管理
final 类或 final 方法CGLIB 无法覆盖 final 方法避免 final 或改设计
代理选择：有接口时可用 JDK 动态代理；没有接口时通常用 CGLIB 生成子类代理。Spring Boot 默认更倾向使用类代理以减少接口限制，但机制仍是代理。

5. 事务机制：@Transactional 的边界和坑

Spring 声明式事务本质上也是 AOP 代理：方法执行前开启事务，成功提交，异常回滚。它的坑基本都来自“没走代理”和“异常规则不符合预期”。

5.1 正确的下单事务边界

@Service
public class OrderApplicationService {
    private final OrderRepository orderRepository;
    private final OutboxRepository outboxRepository;

    public OrderApplicationService(OrderRepository orderRepository,
                                   OutboxRepository outboxRepository) {
        this.orderRepository = orderRepository;
        this.outboxRepository = outboxRepository;
    }

    @Transactional
    public Long createOrder(CreateOrderCommand command) {
        Order order = Order.create(command.userId(), command.items());
        orderRepository.save(order);

        OutboxMessage message = OutboxMessage.of(
            "OrderCreated",
            order.getId().toString(),
            JsonUtils.toJson(new OrderCreatedEvent(order.getId()))
        );
        outboxRepository.save(message);
        return order.getId();
    }
}

业务表和 outbox 消息表同事务提交，避免“订单成功但消息没发出去”。消息投递由后台任务或 CDC 异步完成。

5.2 事务传播

传播行为含义常见使用
REQUIRED有事务就加入，没有就新建默认，最常见
REQUIRES_NEW挂起外层事务，新建独立事务审计日志、独立流水，但要谨慎
NESTED嵌套事务，依赖保存点局部回滚
SUPPORTS有事务就加入，没有就非事务执行只读查询可见
MANDATORY必须存在事务，否则报错强制上游控制事务

5.3 常见失效

@Service
public class UserService {
    public void outer() {
        inner(); // 同类内部调用，没走代理，事务可能不生效
    }

    @Transactional
    public void inner() {
        // update db
    }
}

5.4 异常回滚规则

@Transactional(rollbackFor = Exception.class)
public void importData(MultipartFile file) throws IOException {
    // 默认只对 RuntimeException 和 Error 回滚
    // 如果要 checked exception 回滚，需要 rollbackFor
}

5.5 事务里不要做什么

  • 不要在事务内做长时间 RPC，容易拖长锁持有时间。
  • 不要在事务内发不可回滚的外部消息，优先 outbox。
  • 不要吞掉异常后还期待事务回滚。
  • 不要把过大的批处理放进一个大事务。

6. Spring Boot 自动配置：约定优于配置

Spring Boot 的价值是用 starter、自动配置、外部化配置和 Actuator 快速搭建生产级应用。官方描述里强调“stand-alone、production-grade、opinionated、minimum fuss”。

6.1 自动配置大致流程

  1. `@SpringBootApplication` 组合了配置类、组件扫描和自动配置能力。
  2. Boot 读取自动配置候选类。
  3. 通过 `@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty` 等条件判断是否生效。
  4. 如果 classpath、配置属性、已有 Bean 满足条件，就注册默认 Bean。
  5. 用户自己定义 Bean 时，通常通过 `@ConditionalOnMissingBean` 覆盖默认配置。

6.2 自定义 starter 思路

@ConfigurationProperties(prefix = "audit")
public record AuditProperties(boolean enabled, String appName) {}

public class AuditClient {
    private final AuditProperties properties;

    public AuditClient(AuditProperties properties) {
        this.properties = properties;
    }

    public void send(String action) {
        System.out.println(properties.appName() + ":" + action);
    }
}
@AutoConfiguration
@EnableConfigurationProperties(AuditProperties.class)
@ConditionalOnProperty(prefix = "audit", name = "enabled", havingValue = "true")
public class AuditAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public AuditClient auditClient(AuditProperties properties) {
        return new AuditClient(properties);
    }
}

6.3 自动配置常用条件

条件注解含义例子
@ConditionalOnClassclasspath 有某个类才生效有 Redis 客户端才配 Redis
@ConditionalOnMissingBean用户没自定义 Bean 才生效默认 ObjectMapper
@ConditionalOnProperty配置开关满足才生效feature.enabled=true
@ConditionalOnWebApplicationWeb 应用才生效MVC 配置

7. Spring MVC：请求如何到 Controller

MVC 面试按 DispatcherServlet 主线讲：请求进来，找 Handler，执行拦截器，参数解析，调用 Controller，返回值处理，视图或 JSON 输出。

入口DispatcherServlet 接收请求。
映射HandlerMapping 找到 Controller 方法。
适配HandlerAdapter 负责调用方法。
参数ArgumentResolver 解析路径、请求体、参数。
返回ReturnValueHandler 处理 JSON、视图、响应体。
异常ExceptionHandler / ControllerAdvice 统一处理。

统一响应和异常处理

public record ApiResult<T>(int code, String message, T data) {
    public static <T> ApiResult<T> ok(T data) {
        return new ApiResult<>(0, "ok", data);
    }

    public static <T> ApiResult<T> fail(String message) {
        return new ApiResult<>(500, message, null);
    }
}

@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(BizException.class)
    public ApiResult<Void> handleBiz(BizException e) {
        return ApiResult.fail(e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public ApiResult<Void> handleUnknown(Exception e) {
        return ApiResult.fail("system error");
    }
}

拦截器实战：traceId

@Component
public class TraceInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) {
        String traceId = Optional.ofNullable(request.getHeader("X-Trace-Id"))
            .orElse(UUID.randomUUID().toString());
        MDC.put("traceId", traceId);
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler,
                                Exception ex) {
        MDC.clear();
    }
}

8. 数据访问：Repository、事务和查询边界

Spring Data JPA / MyBatis / JdbcTemplate 都能用，关键是分清 Repository、Service、事务和 DTO 边界。

Repository

封装数据访问，不放业务流程。接口可以是 JPA Repository、MyBatis Mapper 或自定义 DAO。

Service

承载业务事务、领域规则、调用编排。`@Transactional` 通常放在应用服务层。

Controller

处理 HTTP 参数和响应，不直接写复杂业务和事务。

分页查询案例

@RestController
@RequestMapping("/orders")
public class OrderController {
    private final OrderQueryService orderQueryService;

    public OrderController(OrderQueryService orderQueryService) {
        this.orderQueryService = orderQueryService;
    }

    @GetMapping
    public Page<OrderView> page(@RequestParam Long userId,
                                @PageableDefault(size = 20) Pageable pageable) {
        return orderQueryService.pageUserOrders(userId, pageable);
    }
}
@Service
public class OrderQueryService {
    private final OrderRepository orderRepository;

    public OrderQueryService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @Transactional(readOnly = true)
    public Page<OrderView> pageUserOrders(Long userId, Pageable pageable) {
        return orderRepository.findByUserId(userId, pageable)
            .map(order -> new OrderView(order.getId(), order.getStatus(), order.getAmount()));
    }
}

9. Spring 里的设计模式

Spring 本身就是设计模式的大型实战场：容器是工厂，AOP 是代理，JdbcTemplate 是模板方法，ApplicationEvent 是观察者，BeanPostProcessor 是责任链味道的扩展点。

设计模式Spring 体现实战用法
工厂模式BeanFactory、ApplicationContext、FactoryBean统一创建对象，屏蔽实例化细节
单例模式默认 singleton Bean无状态 Service 单例复用
代理模式AOP、事务、缓存、安全不改业务代码增强能力
模板方法JdbcTemplate、RedisTemplate、RestTemplate固定资源管理流程，开放业务回调
策略模式注入 `Map<String, Handler>`多支付渠道、多消息处理器
观察者模式ApplicationEventPublisher / Listener订单创建后发通知、积分、日志
适配器模式HandlerAdapterMVC 适配不同 Handler 调用方式
责任链模式Filter、Interceptor、BeanPostProcessor 链请求链路、校验链、审批链

@Component
public class RemoteClientFactoryBean implements FactoryBean<RemoteClient> {
    @Override
    public RemoteClient getObject() {
        return new RemoteClient("https://api.example.com");
    }

    @Override
    public Class<?> getObjectType() {
        return RemoteClient.class;
    }
}

10. 实战案例：订单下单服务怎么设计

目标：下单要落库、扣库存、发消息、可追踪、失败可补偿。核心原则是短事务、状态机、outbox、幂等。

10.1 Controller 只做入口

@RestController
@RequestMapping("/api/orders")
public class OrderController {
    private final OrderApplicationService orderService;

    public OrderController(OrderApplicationService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    public ApiResult<Long> create(@Valid @RequestBody CreateOrderCommand command) {
        return ApiResult.ok(orderService.createOrder(command));
    }
}

10.2 Service 控制事务边界

@Service
public class OrderApplicationService {
    private final InventoryService inventoryService;
    private final OrderRepository orderRepository;
    private final OutboxRepository outboxRepository;

    public OrderApplicationService(InventoryService inventoryService,
                                   OrderRepository orderRepository,
                                   OutboxRepository outboxRepository) {
        this.inventoryService = inventoryService;
        this.orderRepository = orderRepository;
        this.outboxRepository = outboxRepository;
    }

    @Transactional
    public Long createOrder(CreateOrderCommand command) {
        inventoryService.lockStock(command.skuId(), command.count());

        Order order = Order.create(command.userId(), command.skuId(), command.count());
        orderRepository.save(order);

        outboxRepository.save(OutboxMessage.orderCreated(order.getId()));
        return order.getId();
    }
}

10.3 Outbox 后台投递

@Component
public class OutboxPublisher {
    private final OutboxRepository outboxRepository;
    private final MessageClient messageClient;

    public OutboxPublisher(OutboxRepository outboxRepository, MessageClient messageClient) {
        this.outboxRepository = outboxRepository;
        this.messageClient = messageClient;
    }

    @Scheduled(fixedDelay = 1000)
    public void publish() {
        List<OutboxMessage> messages = outboxRepository.findPending(100);
        for (OutboxMessage message : messages) {
            try {
                messageClient.send(message.topic(), message.body());
                outboxRepository.markSent(message.id());
            } catch (Exception e) {
                outboxRepository.markFailed(message.id(), e.getMessage());
            }
        }
    }
}
这个案例里用到的模式：Service Layer、Repository、状态机、Outbox、模板式调度、事务代理。面试可以把它和 MQ 可靠性、幂等、补偿一起讲。

11. 实战案例：支付渠道策略模式

多渠道支付别写一堆 if/else。让每个支付渠道实现同一个接口，Spring 自动注入成 `Map`，路由时按渠道选择策略。

11.1 策略接口

public interface PayStrategy {
    String channel();
    PayResult pay(PayCommand command);
}

11.2 两个实现

@Component
public class AlipayStrategy implements PayStrategy {
    @Override
    public String channel() {
        return "ALIPAY";
    }

    @Override
    public PayResult pay(PayCommand command) {
        return PayResult.success("alipay-trade-no");
    }
}

@Component
public class WechatPayStrategy implements PayStrategy {
    @Override
    public String channel() {
        return "WECHAT";
    }

    @Override
    public PayResult pay(PayCommand command) {
        return PayResult.success("wechat-trade-no");
    }
}

11.3 路由器

@Service
public class PayRouter {
    private final Map<String, PayStrategy> strategies;

    public PayRouter(List<PayStrategy> strategyList) {
        this.strategies = strategyList.stream()
            .collect(Collectors.toUnmodifiableMap(PayStrategy::channel, Function.identity()));
    }

    public PayResult pay(PayCommand command) {
        PayStrategy strategy = strategies.get(command.channel());
        if (strategy == null) {
            throw new BizException("unsupported pay channel: " + command.channel());
        }
        return strategy.pay(command);
    }
}

11.4 为什么比 if/else 好

  • 新增渠道只新增一个策略类，不改核心路由逻辑。
  • 每个渠道可独立测试、独立配置、独立降级。
  • 符合开闭原则，也方便按渠道做 AOP 监控。

12. 实战案例：ApplicationEvent 做领域事件解耦

订单创建后可能要发优惠券、加积分、发通知、写审计。不要把这些都塞进下单主流程，可以用事件解耦；强可靠跨进程仍优先 MQ/outbox。

12.1 发布事件

public record OrderCreatedEvent(Long orderId, Long userId) {}

@Service
public class OrderDomainService {
    private final ApplicationEventPublisher publisher;

    public OrderDomainService(ApplicationEventPublisher publisher) {
        this.publisher = publisher;
    }

    @Transactional
    public Long createOrder(CreateOrderCommand command) {
        Long orderId = saveOrder(command);
        publisher.publishEvent(new OrderCreatedEvent(orderId, command.userId()));
        return orderId;
    }
}

12.2 事务提交后再处理

@Component
public class CouponListener {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onOrderCreated(OrderCreatedEvent event) {
        // 只有下单事务提交成功后才发券
        System.out.println("send coupon to user " + event.userId());
    }
}

12.3 异步监听

@Configuration
@EnableAsync
class AsyncConfig {
}

@Component
public class NotifyListener {
    @Async
    @EventListener
    public void notifyUser(OrderCreatedEvent event) {
        // 异步发送通知
    }
}
注意：Spring 本地事件只在当前进程内可靠。跨服务、强可靠通知要用 MQ/outbox；本地事件适合进程内模块解耦。

13. 实战案例：缓存、幂等与切面

Spring 的缓存抽象适合读多写少；幂等可以用 AOP 包装接口入参和业务 key，统一做重复提交控制。

13.1 Cacheable 查询缓存

@Service
public class ProductQueryService {
    private final ProductRepository productRepository;

    public ProductQueryService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    @Cacheable(cacheNames = "product", key = "#skuId", unless = "#result == null")
    public ProductView getProduct(Long skuId) {
        return productRepository.findBySkuId(skuId)
            .map(ProductView::from)
            .orElse(null);
    }

    @CacheEvict(cacheNames = "product", key = "#skuId")
    public void updateProduct(Long skuId, UpdateProductCommand command) {
        productRepository.update(skuId, command);
    }
}

13.3 幂等切面

@Aspect
@Component
public class IdempotentAspect {
    private final StringRedisTemplate redisTemplate;

    public IdempotentAspect(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    @Around("@annotation(idempotent)")
    public Object around(ProceedingJoinPoint pjp, Idempotent idempotent) throws Throwable {
        String key = parseKey(pjp, idempotent.key());
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", Duration.ofSeconds(idempotent.ttlSeconds()));
        if (!Boolean.TRUE.equals(success)) {
            throw new BizException("duplicate request");
        }
        return pjp.proceed();
    }

    private String parseKey(ProceedingJoinPoint pjp, String expression) {
        return "idem:" + expression;
    }
}

15. 高频面试回答模板

Q1：Spring IoC 是什么？
IoC 是控制反转，对象不再自己创建依赖，而是由 Spring 容器负责创建、装配和管理生命周期。DI 是实现 IoC 的主要方式，Spring 通过构造器、setter、字段等方式把依赖注入 Bean。这样可以降低耦合、方便替换实现、测试和做 AOP 增强。
Q2：Bean 生命周期怎么讲？
先把配置解析成 BeanDefinition，然后实例化 Bean，填充属性，执行 Aware 回调，执行 BeanPostProcessor 前置处理，执行初始化方法，比如 @PostConstruct 和 InitializingBean，再执行 BeanPostProcessor 后置处理，AOP 代理通常在后置处理阶段生成。容器关闭时执行销毁回调。
Q3：@Transactional 为什么会失效？
因为声明式事务基于代理。常见失效包括同类内部方法调用没走代理、方法不是 public、对象不是 Spring Bean、异常被吞掉、默认 checked exception 不回滚、数据库引擎不支持事务、异步线程里事务上下文不传播等。
Q4：Spring Boot 自动配置原理？
Boot 根据 classpath、配置属性、已有 Bean、Web 环境等条件判断是否注册默认 Bean。它通过 starter 引入依赖，通过自动配置类提供默认配置，通过条件注解控制是否生效。用户自定义 Bean 时，很多默认配置会因为 @ConditionalOnMissingBean 而让位。
Q5：Spring 用到了哪些设计模式？
BeanFactory/ApplicationContext 是工厂模式，默认 singleton Bean 体现单例，AOP 和事务是代理模式，JdbcTemplate 是模板方法，ApplicationEvent 是观察者，HandlerAdapter 是适配器，Filter/Interceptor 是责任链，业务多渠道处理常用策略模式。
Q6：如何设计多支付渠道？
定义 PayStrategy 接口，每个渠道实现一个策略 Bean。启动时注入 List 或 Map，构建 channel 到 strategy 的映射。支付时根据渠道路由到对应策略。这样新增渠道只加实现类，不改核心流程，也方便按渠道做监控、降级和测试。
Q7：本地事件和 MQ 怎么选？
Spring ApplicationEvent 适合进程内模块解耦，比如订单创建后本应用内发通知、写审计。跨服务或强可靠场景不要只靠本地事件，应该用 MQ 或 outbox，保证消息可重试、可追踪、可补偿。
