2. 常用设计模式详解，结合 Spring 源码举例，手写线程安全单例

设计模式不是为了炫技，而是为了控制变化。面试重点是“这个模式解决了什么变化点”。

模式解决的问题Spring 中的影子面试表达
单例模式一个类只需要一个实例，避免重复创建和状态分裂。Spring 默认 singleton Bean。单例强调对象唯一，Spring 单例强调容器内唯一。
工厂模式把对象创建和使用解耦。BeanFactory、FactoryBean。使用方不关心怎么 new，只关心拿到什么能力。
代理模式不改目标类，为方法增加增强逻辑。AOP、事务代理。事务、日志、权限都适合代理增强。
模板方法固定主流程，把变化步骤交给子类或回调。JdbcTemplate、RestTemplate、TransactionTemplate。资源获取、异常处理、释放资源由框架统一做。
观察者模式事件发生后通知多个订阅者。ApplicationEvent、ApplicationListener、@EventListener。主流程和后置动作解耦。
适配器模式把不同接口统一成框架认识的调用方式。HandlerAdapter。Controller 写法很多，Spring MVC 用适配器统一调用。
责任链模式请求经过一串处理器，每个处理器只负责一部分。FilterChain、Interceptor、Spring Security FilterChain。鉴权、日志、限流、灰度都可以链式处理。
策略模式同一个业务点有多种算法/规则可替换。多个 Handler Bean 自动注入 List/Map。消灭 if-else，让新增规则变成新增类。

线程安全单例 1：饿汉式

public class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() {
        return INSTANCE;
    }
}

类加载时创建，天然线程安全。缺点是不能懒加载，如果对象很重或者未必会用，就会提前占资源。

线程安全单例 2：synchronized 懒汉式

public class Singleton {
    private static Singleton instance;
    private Singleton() {}
    public static synchronized Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}

懒加载、线程安全，但每次获取都加锁，性能一般。

线程安全单例 3：双重检查锁 DCL

public class Singleton {
    private static volatile Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
为什么必须 volatile？ new Singleton() 不是原子操作，可能分为分配内存、初始化对象、引用指向内存。没有 volatile，指令重排可能导致其他线程看到一个“引用非空但对象未初始化完成”的实例。

线程安全单例 4：静态内部类

public class Singleton {
    private Singleton() {}
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}

推荐口径：懒加载、线程安全、实现简单。外部类加载时不会初始化 Holder，第一次调用 getInstance 才触发 Holder 类加载。

线程安全单例 5：枚举

public enum Singleton {
    INSTANCE;
    public void execute() {
        System.out.println("do something");
    }
}

枚举单例能天然防反射和反序列化破坏，是 Java 层面最稳的单例写法。实际业务里如果需要继承、延迟初始化或 Spring 管理，仍然会用普通 Bean。

3. 策略、工厂、观察者在业务中的落地，如何用设计模式解耦代码？

最好用“支付方式 / 订单处理 / 优惠计算 / 风控规则”举例，因为面试官非常容易理解。

反例：所有业务堆在 if-else

public void pay(Order order) {
    if ("ALI".equals(order.getPayType())) {
        // 支付宝逻辑
    } else if ("WECHAT".equals(order.getPayType())) {
        // 微信逻辑
    } else if ("BANK".equals(order.getPayType())) {
        // 银行卡逻辑
    }
}

问题：新增一种支付方式就修改老代码，主流程越来越胖，测试范围变大，违反开闭原则。

策略模式：每种支付一种策略

public interface PayStrategy {
    String type();
    PayResult pay(Order order);
}

@Component
public class AliPayStrategy implements PayStrategy {
    public String type() { return "ALI"; }
    public PayResult pay(Order order) {
        // 支付宝支付逻辑
        return PayResult.success();
    }
}

工厂模式：自动收集策略 Bean

@Component
public class PayStrategyFactory {
    private final Map<String, PayStrategy> strategyMap = new ConcurrentHashMap<>();

    public PayStrategyFactory(List<PayStrategy> strategies) {
        for (PayStrategy strategy : strategies) {
            strategyMap.put(strategy.type(), strategy);
        }
    }

    public PayStrategy get(String type) {
        PayStrategy strategy = strategyMap.get(type);
        if (strategy == null) {
            throw new IllegalArgumentException("unsupported pay type: " + type);
        }
        return strategy;
    }
}

观察者模式：支付成功后的动作解耦

public record PaySuccessEvent(Long orderId, Long userId) {}

@Service
public class PayService {
    private final ApplicationEventPublisher publisher;
    public void pay(Order order) {
        // 1. 支付主流程
        // 2. 更新订单状态
        publisher.publishEvent(new PaySuccessEvent(order.getId(), order.getUserId()));
    }
}

@Component
public class CouponListener {
    @EventListener
    public void onPaySuccess(PaySuccessEvent event) {
        // 发优惠券
    }
}
高级补充：如果后置动作必须等事务提交后再执行，可以使用 @TransactionalEventListener(phase = AFTER_COMMIT)。否则主事务回滚了，监听器却发了券或发了消息，就会产生业务不一致。
面试总结：策略模式解决“多种处理方式”的扩展问题；工厂模式解决“怎么选择策略”的问题；观察者模式解决“主流程和后置动作解耦”的问题。落地到 Spring 时，可以利用 Bean 自动注入 List/Map、ApplicationEvent 和 AOP，把新增业务变成新增类，而不是不断修改主流程。

4. Spring Bean 完整生命周期、后置处理器扩展机制，项目中如何自定义扩展 Spring？

Spring 扩展的本质：框架把启动、实例化、初始化、代理、销毁这些关键节点开放给你。

1. 扫描
解析 XML、注解、配置类。
2. BeanDefinition
记录 class、scope、依赖、初始化方法。
3. 实例化
构造方法创建原始对象。
4. 属性填充
@Autowired、@Value 注入。
5. Aware
回调 BeanName、BeanFactory、ApplicationContext。
6. BPP before
初始化前置处理。
7. init
@PostConstruct、afterPropertiesSet、init-method。
8. BPP after
初始化后处理，AOP 代理常在这里生成。
9. 使用
容器返回最终 Bean，可能是代理。
10. 销毁
@PreDestroy、DisposableBean、destroy-method。

BeanPostProcessor：修改 Bean 实例

@Component
public class HandlerRegistryPostProcessor implements BeanPostProcessor {
    private final Map<String, BizHandler> handlers = new ConcurrentHashMap<>();

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (bean instanceof BizHandler handler) {
            handlers.put(handler.type(), handler);
        }
        return bean;
    }

    public BizHandler get(String type) {
        return handlers.get(type);
    }
}

适合场景：自动收集策略、启动期校验注解、为特定 Bean 包代理、注册插件、做通用能力增强。

BeanFactoryPostProcessor：修改 BeanDefinition

@Component
public class MyBeanFactoryPostProcessor implements BeanFactoryPostProcessor {
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        BeanDefinition bd = beanFactory.getBeanDefinition("orderService");
        bd.setLazyInit(true);
    }
}

它发生在 Bean 实例化之前，处理的是 BeanDefinition，不是 Bean 实例。

自定义 Spring 扩展落地案例

扩展点项目落地说明
AOP + 自定义注解幂等、操作日志、接口耗时、权限校验。方法级增强，不侵入业务方法。
BeanPostProcessor启动时自动收集 Handler/Strategy。消除手工注册 Map。
ApplicationListener订单成功后发消息、发通知、写审计。主流程与后置动作解耦。
FactoryBean封装复杂客户端，如 RPC client、规则引擎 client。容器里看到的是产品对象，不是工厂本身。
AutoConfiguration封装公司内部 starter。通过条件装配把通用能力一键接入业务系统。
60 秒回答：Spring Bean 生命周期从 BeanDefinition 注册开始，经过实例化、属性填充、Aware 回调、BeanPostProcessor 前置处理、初始化方法、BeanPostProcessor 后置处理，最后进入使用和销毁阶段。AOP 代理通常在后置处理器阶段生成。项目中如果要扩展 Spring，可以用 AOP 做方法增强，用 BeanPostProcessor 收集或替换 Bean，用 BeanFactoryPostProcessor 修改 BeanDefinition，用事件机制解耦后置动作，也可以把通用能力封装成 starter，通过自动配置和条件装配复用。

5. Spring 事务传播机制：嵌套事务、挂起事务底层逻辑与实战踩坑

事务传播回答的核心：当一个事务方法调用另一个事务方法时，内层方法如何处理当前已经存在的事务。

传播行为含义典型场景
REQUIRED有事务加入，没有事务新建。默认。普通业务主流程。
REQUIRES_NEW挂起当前事务，新开一个独立事务。操作日志、审计记录、失败记录。
NESTED同一个物理事务内创建 savepoint。局部失败可回滚，不一定影响外层。
SUPPORTS有事务就加入，没有就非事务执行。可有可无的读操作。
NOT_SUPPORTED以非事务方式执行，有事务则挂起。不希望占事务的慢查询、外部调用。
MANDATORY必须存在事务，否则报错。强制被主流程包裹。
NEVER必须没有事务，否则报错。明确禁止事务的操作。

底层拦截主线

代理对象拦截外部调用进入事务代理，TransactionInterceptor 开始工作。
获取事务属性读取 @Transactional 的传播行为、隔离级别、超时、rollbackFor。
判断当前线程是否已有事务通过 TransactionSynchronizationManager 绑定的连接、同步器判断。
根据传播行为处理REQUIRED 加入；REQUIRES_NEW 挂起旧事务并申请新连接；NESTED 创建 savepoint。
执行业务方法正常返回则提交，异常则按 rollback 规则回滚。
恢复现场如果之前挂起了外层事务，内层结束后恢复外层事务上下文。

REQUIRES_NEW：挂起外层，新开内层

@Transactional
public void createOrder() {
    orderMapper.insert(order);       // T1
    auditService.saveAuditLog(order); // T2, REQUIRES_NEW
    throw new RuntimeException();     // T1 rollback, T2 committed
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveAuditLog(Order order) {
    auditMapper.insert(order.toAuditLog());
}

执行逻辑：外层 T1 开启后，调用内层时 T1 被挂起；内层申请新连接开启 T2；T2 提交后恢复 T1；T1 后续异常只回滚 T1，不影响已经提交的 T2。

连接池坑：REQUIRES_NEW 需要额外数据库连接。如果大量请求外层事务持有连接，同时内层事务又申请新连接，连接池可能耗尽。高并发场景要谨慎。

NESTED：同一物理事务里的 savepoint

@Transactional
public void createOrder() {
    orderMapper.insert(order);
    try {
        stockService.deduct(order); // NESTED
    } catch (Exception e) {
        // 内层回滚到保存点，外层可以继续
    }
    orderMapper.updateStatus(order.getId(), "PARTIAL_SUCCESS");
}

@Transactional(propagation = Propagation.NESTED)
public void deduct(Order order) {
    stockMapper.deduct(order.getSkuId());
    throw new RuntimeException();
}

NESTED 不一定新开物理事务，而是在当前事务中创建保存点。内层失败可以回滚到 savepoint，外层决定继续提交还是整体回滚。

事务失效经典坑

坑为什么解决
同类内部调用this.inner() 绕过代理。拆到另一个 Service，或注入代理对象。
异常被 catch代理感知不到异常。重新抛出，或 setRollbackOnly。
默认只回滚 RuntimeException受检异常默认不回滚。@Transactional(rollbackFor = Exception.class)
方法不是 public代理无法按预期增强。事务方法保持 public。
数据库引擎不支持事务比如 MyISAM。使用 InnoDB。
异步线程里使用事务上下文事务资源绑定在线程本地变量。异步方法单独开事务，别假设继承外层事务。

6. 微服务服务注册与发现完整流程，Nacos 核心原理与心跳机制

回答注册中心要说完整闭环：注册、心跳、健康检查、服务发现、本地缓存、变更推送、故障摘除。

服务注册流程

  1. 服务启动，读取服务名、IP、端口、集群、权重、元数据。
  2. Nacos Client 向 Nacos Server 注册实例。
  3. Nacos Server 保存服务与实例映射。
  4. 临时实例定期心跳，续约存活状态。
  5. 服务停止或心跳过期，实例被标记不健康或剔除。

服务发现流程

  1. 消费者根据服务名拉取实例列表。
  2. 客户端本地缓存可用实例。
  3. 结合健康状态、权重、同集群优先进行负载均衡。
  4. 服务列表变化时，Nacos 推送或客户端轮询更新。
  5. 调用失败时结合重试、熔断、摘除做兜底。

心跳机制

Service Instance:
  serviceName = order-service
  ip = 10.1.2.3
  port = 8080
  ephemeral = true
  metadata = { version: v1, zone: xa }

Client -> Server heartbeat every few seconds
Server updates lastBeat
if timeout: mark unhealthy
if long timeout: remove ephemeral instance

临时实例依赖心跳维持存活，适合普通微服务实例。持久实例更强调元数据持久化，即使客户端不在线也不会立刻消失，通常适合对服务实例管理更强约束的场景。

AP / CP 怎么答

表达：Nacos 在服务发现里通常需要优先保证可用性，因为实例列表短暂不一致可以通过重试、负载均衡和健康检查兜底；但配置、持久实例等场景会更强调一致性。注册中心选型本质是在一致性、可用性和故障恢复速度之间取舍。

故障场景

问题影响兜底
服务实例宕机消费者可能短时间拿到旧实例。心跳摘除、客户端失败重试、熔断。
Nacos Server 短暂不可用新注册/配置拉取受影响。客户端本地缓存继续调用旧列表。
网络分区不同节点看到的实例列表不一致。最终一致、健康检查、重试和限流。
大量实例上下线推送风暴、客户端缓存频繁刷新。批量推送、延迟合并、客户端限频。

7. 架构设计题：用户登录在线时长统计系统，精准统计每日在线峰值

这题考你能不能拆指标。在线时长和在线峰值不是一回事：一个是用户维度累计，一个是全站维度最大并发在线数。

需求澄清

指标定义难点
用户每日在线时长用户当天处于在线状态的总时长。跨天切分、异常断开、重复登录、多端登录。
每日在线峰值当天任意时刻全站在线用户数最大值。高频心跳、去重、过期清理、原子更新峰值。
实时在线人数当前未超时的在线用户数。心跳延迟、Redis 热点、统计成本。

基础方案：登录 + 心跳 + 退出 + 超时

LOGIN(userId, sessionId, ts)
HEARTBEAT(userId, sessionId, ts)
LOGOUT(userId, sessionId, ts)
TIMEOUT(userId, sessionId, ts)

不能只靠登录和退出，因为用户可能直接关浏览器、断网、杀进程。所以必须依赖心跳和 TTL 超时兜底。

Redis 数据结构

# 当前在线 session，TTL 例如 5 分钟
online:session:{sessionId} = {userId, loginTime, lastHeartbeatTime}

# 用户多端在线集合
online:user:{userId} = Set(sessionId...)

# 全站在线集合，score 是最后心跳时间
zset:online:sessions = ZSET(sessionId -> lastHeartbeatTime)

# 每日峰值
daily:online:peak:{yyyyMMdd} = maxOnlineCount

当前在线人数与峰值

// 每次心跳：
ZADD zset:online:sessions now sessionId
EXPIRE online:session:{sessionId} 300

// 统计当前在线：
ZREMRANGEBYSCORE zset:online:sessions 0 now - timeout
current = ZCARD zset:online:sessions

// 更新每日峰值：
old = GET daily:online:peak:20260629
if current > old:
    SET daily:online:peak:20260629 current
原子性：清理过期、统计当前在线、更新峰值最好用 Lua 脚本或单线程消费器串行完成，避免并发更新峰值时丢最大值。

边界问题

  • 多端登录：按 session 统计在线，按 userId 去重统计在线用户，口径要提前定义。
  • 重复登录：同一个 sessionId 幂等覆盖，不重复累计。
  • 异常断开：用最后心跳 + timeout 作为结束时间。
  • 心跳乱序：只接受大于当前 lastHeartbeat 的时间。
  • 跨天：按自然日切片累加。
  • 峰值精度：如果要求秒级精准，事件流聚合；如果允许分钟级，定时采样也可以。
60 秒回答：我会把系统拆成在线状态、在线时长和每日峰值三部分。在线状态用 Redis 维护 session，登录写入，心跳续期，退出删除，异常断开靠 TTL 超时兜底。在线时长基于 session 的开始和结束时间计算，跨天要切分，多端要明确按用户去重还是按 session 统计。每日峰值可以用 Redis ZSet 存活跃 session，score 是最后心跳时间，定期清理过期 session，计算当前在线数，并用 Lua 原子更新当天最大值。高并发下把登录、心跳、退出事件写入 MQ，由实时计算任务聚合，Redis 支撑实时查询，MySQL 或 ClickHouse 保存历史报表。

大流量多级兜底话术

大流量系统不能只靠一个限流点。我会做分层防线：入口层通过 CDN、WAF 和网关做流量清洗与粗限流；应用层按用户、接口、租户、热点资源做细粒度限流；服务调用层设置超时、重试、熔断、线程池隔离；数据层通过缓存、本地缓存、热点 key 拆分、读写分离和异步削峰保护数据库。核心链路优先保障，非核心链路允许降级为空数据、静态数据或延迟处理。所有策略都要可配置、可观测、可灰度，避免硬编码导致故障时无法快速调整。

重试风暴必须主动提

重试不是越多越好。下游已经慢或已经挂了，上游无脑重试会把流量放大，形成雪崩。写请求还要考虑幂等，否则可能重复下单、重复扣款。

Retry rule:
  maxAttempts = 2 or 3
  backoff = exponential + jitter
  only retry idempotent operation
  combine with timeout + circuit breaker

高可用

系统少宕机，宕机能自动恢复。手段：多副本、负载均衡、故障转移、限流、熔断、降级、灰度、备份。

Throughput

业务增长或需求变化时低成本扩容和演进。手段：模块化、接口抽象、策略化、插件化、无状态、水平扩容。

从单体到微服务怎么拆

  • 先按业务边界拆，不要按技术层拆。
  • 高频变化和低频稳定模块分开。
  • 读多写少和交易写链路分开。
  • 强一致链路不要过度拆分，否则分布式事务成本很高。
  • 先模块化，再服务化，避免为了微服务而微服务。

10. 算法手撕：旋转数组找最小值，二分查找进阶

这题核心不是背代码，而是说清楚为什么比较 nums[mid] 和 nums[right]。

无重复元素

public int findMin(int[] nums) {
    int left = 0, right = nums.length - 1;
    while (left < right) {
        int mid = left + (right - left) / 2;
        if (nums[mid] > nums[right]) {
            left = mid + 1;
        } else {
            right = mid;
        }
    }
    return nums[left];
}

解释：旋转数组由两个有序段组成。如果 nums[mid] > nums[right]，说明 mid 在左侧大段，最小值一定在 mid 右边；否则 mid 在右侧小段或数组未旋转，最小值在 left 到 mid 之间，right = mid 不能跳过 mid。

有重复元素

public int findMinWithDuplicates(int[] nums) {
    int left = 0, right = nums.length - 1;
    while (left < right) {
        int mid = left + (right - left) / 2;
        if (nums[mid] > nums[right]) {
            left = mid + 1;
        } else if (nums[mid] < nums[right]) {
            right = mid;
        } else {
            right--;
        }
    }
    return nums[left];
}

当 nums[mid] == nums[right] 时无法判断最小值在哪边，只能保守缩小右边界。平均 O(log n)，最坏 O(n)，例如全是重复元素。

Rust 版本

pub fn find_min(nums: Vec<i32>) -> i32 {
    let mut left = 0usize;
    let mut right = nums.len() - 1;
    while left < right {
        let mid = left + (right - left) / 2;
        if nums[mid] > nums[right] {
            left = mid + 1;
        } else {
            right = mid;
        }
    }
    nums[left]
}

pub fn find_min_with_duplicates(nums: Vec<i32>) -> i32 {
    let mut left = 0usize;
    let mut right = nums.len() - 1;
    while left < right {
        let mid = left + (right - left) / 2;
        if nums[mid] > nums[right] {
            left = mid + 1;
        } else if nums[mid] < nums[right] {
            right = mid;
        } else {
            right -= 1;
        }
    }
    nums[left]
}

面试官追问

追问回答
为什么不是和 left 比？也可以设计，但和 right 比更容易判断右侧有序段和最小值方向，边界更稳。
为什么 right = mid，不是 mid - 1？因为 mid 可能就是最小值，不能丢掉。
有重复为什么会退化？相等时无法判断方向，只能 right--，极端重复数组每次缩一位。

最后：10 题口述模板

1. 自我介绍：我会从复杂业务链路、性能优化、工程效率三块介绍自己，并强调我关注状态流转、异常闭环、幂等补偿和可观测性。
2. 设计模式：设计模式本质是控制变化。Spring 里 BeanFactory 是工厂，AOP 是代理，JdbcTemplate 是模板方法，ApplicationEvent 是观察者，HandlerAdapter 是适配器。
3. 业务解耦：策略模式消灭 if-else，工厂统一选择策略，观察者解耦主流程和后置动作。Spring 里可以用 List 注入所有策略，用事件监听做后置扩展。
4. Bean 生命周期：BeanDefinition、实例化、属性填充、Aware、BPP before、init、BPP after、使用、销毁。AOP 代理常在 BPP after 阶段生成。
5. 事务传播：REQUIRED 加入当前事务；REQUIRES_NEW 挂起外层新开内层；NESTED 在同一物理事务创建 savepoint。事务失效常见于内部调用、异常被 catch、非 public、rollbackFor 不对。
6. Nacos：服务启动注册实例，临时实例定时心跳，服务端根据心跳判断健康和剔除，消费者拉取并缓存实例列表，变化时推送或刷新。
7. 在线统计：登录、心跳、退出、超时四类事件；Redis 维护在线状态和 ZSet；在线时长按 session 计算并跨天切分；峰值用当前在线数原子更新。
8. 高并发兜底：限流挡流量，熔断保下游，降级保核心。入口、网关、应用、调用、数据层都要有防线。
9. 架构取舍：高可用关注故障恢复，高并发关注吞吐，可扩展性关注低成本演进。交易链路一致性优先，统计推荐链路性能和可用性优先。
10. 旋转数组：比较 mid 和 right 判断最小值方向。mid 大于 right，最小值在右侧；否则在左侧含 mid；重复元素相等时 right--。
