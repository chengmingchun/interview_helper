订单模块要讲成“状态驱动的业务编排链路”，不是功能点罗列。

面试官追问订单，通常不是想听你写了几个接口，而是想确认你是否真的处理过高并发、幂等、重复入库、缓存一致性、状态流转、异步下发和慢 SQL。回答要从业务链路进入，再用具体案例证明你能定位问题、解释原理、做工程取舍。

项目主线：先把订单的“位置”讲清楚

一上来别讲技术细节，先告诉面试官订单在系统里承担什么责任。

Project Context

推荐开场

我参与的是一个云服务管理编排平台，订单模块处在上游资源申请和下游服务下发之间。它不是单纯保存订单记录，而是把一次资源申请变成可追踪、可重试、可查询的业务流程。链路上会经过网关鉴权、租户和配额校验、订单主记录和明细落库、异步服务下发、状态轮询或回写，最后在查询侧展示最终状态。

你要主动抛出的关键词

  • 订单是业务编排链路，不是 CRUD。
  • 创建侧关注幂等和重复入库。
  • 下发侧关注异步、重试、补偿。
  • 状态侧关注状态机、CAS、乱序回写。
  • 查询侧关注缓存一致性和慢 SQL。

面试官听完会怎么追

  • 那你们怎么防重复提交？
  • 下游超时订单状态怎么处理？
  • 缓存和数据库不一致怎么办？
  • 你说的高并发具体在哪里？
  • SQL 从 49s 到 0.9s 怎么定位的？
订单主链路：
上游请求 -> 网关鉴权 -> 租户/配额校验 -> 参数校验
-> 生成幂等键 -> 订单主记录落库 -> 订单明细落库
-> 写异步下发任务 -> 调用下游服务 -> 状态回写/轮询
-> 查询侧展示最终状态

案例 1：订单创建幂等与重复入库

这是订单模块最高频追问：重复点击、接口超时重试、并发请求会不会创建两笔订单。

Case 1 / Idempotency

怎么组织语言

先讲业务场景

订单创建接口可能因为页面重复点击、上游超时重试、网络抖动或批量提交被多次调用。如果不做幂等，就可能出现重复订单、重复下发、重复扣减配额，后续状态也会变得不可控。

再讲设计方案

创建侧生成业务幂等键，数据库建立唯一约束。第一个请求插入成功，后续重复请求触发唯一键冲突后查询已有订单并返回同一个订单视图。下发任务也携带同一个请求标识，避免下游重复执行。

推荐回答

订单创建这块我重点考虑的是重复提交和超时重试。因为订单创建一旦重复，不只是多一条记录，还可能导致重复下发和状态不一致。所以我不会只靠前端按钮置灰或先查后插，而是会根据租户、外部请求号、资源类型和关键参数摘要生成业务幂等键，并在数据库层加唯一索引兜底。并发情况下只有一个请求能插入成功，另一个请求捕获唯一键冲突后查询已有订单并返回。这样即使应用层锁失效、接口超时重试或者服务重启，也不会重复入库。

@Transactional
public OrderView createOrder(CreateOrderCmd cmd) {
    String idemKey = buildIdemKey(cmd);
    try {
        orderMapper.insert(OrderDO.pending(idemKey, cmd));
        taskMapper.insert(DispatchTask.orderCreated(idemKey));
    } catch (DuplicateKeyException e) {
        return orderMapper.findByIdemKey(idemKey);
    }
    return orderMapper.findByIdemKey(idemKey);
}

基础追问 1：为什么不只用 Redis 锁？

答：Redis 锁可以削峰，但不能作为最终一致性兜底。锁可能过期、释放失败、服务重启，也可能因为锁粒度设计不合理漏掉并发。订单创建最终要靠数据库唯一约束保证“同一个幂等键只能插入一次”。

基础追问 2：为什么先查后插不可靠？

答：两个请求可能同时查不到，然后同时插入。除非加锁或唯一索引，否则先查后插不是原子操作。数据库唯一索引把判断和写入收敛到存储层，能处理真正的并发竞争。

基础追问 3：事务里调用下游吗？

答：不建议在数据库事务里直接调用下游。下游慢会拉长事务，增加锁持有时间和失败复杂度。更好的方式是订单落库和任务落库在一个事务里完成，事务提交后由异步任务下发。

你要避免的坑

  • 不要说“用 synchronized 防重复”，分布式场景不成立。
  • 不要说“前端禁用按钮就行”，接口重试仍然存在。
  • 不要只说“Redis 锁”，要强调唯一索引是兜底。

案例 2：状态机、异步下发与乱序回写

面试官会用这个问题判断你是否理解订单不是一条记录，而是一段生命周期。

Case 2 / State Machine

怎么组织语言

先讲业务场景

订单创建后不会立即完成，下游资源创建可能耗时、失败、超时或回调乱序。订单状态必须能表达“处理中、成功、失败、取消、重试中”等阶段，否则排障时无法判断卡在哪一步。

再讲技术设计

用状态机约束合法流转，用前置状态或版本号做 CAS 更新。下游调用异步化，失败按错误类型重试，长时间处理中订单由补偿任务扫描确认最终状态。

推荐回答

订单状态这块我会按状态机来设计，而不是简单 update 覆盖。比如订单从待处理进入下发中，再进入成功或失败；失败后可以触发重试，取消也有独立流转。更新状态时带前置状态或版本号，保证只有符合预期的状态才能推进。这样即使下游回调和本地轮询同时回来，也不会出现旧消息覆盖新状态。对于下游超时，我不会立即判失败，而是先保持处理中，再通过补偿任务或状态查询确认最终结果。

PENDING -> PROCESSING -> SUCCESS
PENDING -> PROCESSING -> FAILED
FAILED  -> PROCESSING  // 重试
PROCESSING -> CANCELING -> CANCELED

update order
set status = ?, version = version + 1
where order_id = ?
  and status = ?
  and version = ?;

基础追问 1：为什么要版本号？

答：版本号是乐观锁，解决并发更新互相覆盖的问题。只有当前版本和预期一致才更新成功，否则说明状态已经被其他线程推进，需要重新查询后决策。

基础追问 2：回调乱序怎么办？

答：不能后到覆盖先到。要定义合法状态流转和终态优先级，通过前置状态、版本号或状态优先级判断是否允许更新。已经 SUCCESS 的订单不能被旧的 FAILED 回调覆盖。

基础追问 3：补偿任务扫什么？

答：通常扫描长时间处于 PROCESSING、RETRYING、CANCELING 的订单，根据下游查询结果或任务执行记录修正状态。补偿不是盲目重试，而是让状态最终收敛。

你要避免的坑

  • 不要说“失败就直接改 FAILED”，超时可能只是未知态。
  • 不要说“回调来了就 update”，要判断合法流转。
  • 不要忽略取消、重试、补偿这些边界状态。

Case 3 / Cache Consistency

怎么组织语言

先讲业务场景

订单详情和订单状态会被频繁查询，尤其是创建后页面轮询状态。如果每次都打数据库，查询压力会比较大；但如果缓存状态不一致，用户会看到旧状态，比如订单已经成功但页面还显示处理中。

再讲一致性策略

订单状态以数据库为准，缓存用于查询减压。写侧先更新数据库，事务提交后删除缓存；删除失败进入重试，TTL 兜底。状态缓存可以带版本号或更新时间，避免旧值回填覆盖新值。

推荐回答

订单缓存我会定位成查询侧加速，而不是准源。写状态时先更新数据库，事务提交后删除缓存，避免事务未提交就让读请求穿透到旧数据。删除缓存失败会进入重试队列，缓存本身设置较短 TTL 兜底。对于订单状态这类会频繁变化的数据，我会尽量缓存详情或热点状态，并在回填时带更新时间或版本判断，避免旧状态覆盖新状态。

读流程：cache -> DB -> 回填 cache + TTL
写流程：update DB -> commit -> delete cache -> 删除失败重试
兜底：短 TTL + 重试队列 + 版本/更新时间判断

基础追问 1：为什么不是先删缓存再更新 DB？

答：先删缓存后更新 DB，中间如果有读请求进来，可能读到旧 DB 并回填旧缓存，导致脏数据停留更久。先更新 DB，提交后删缓存更常见，再配合重试和 TTL 兜底。

基础追问 2：删除缓存失败怎么办？

答：不能静默失败。可以记录重试任务，或者通过消息/binlog 异步删除。TTL 是最终兜底，但不能只靠 TTL，否则旧状态可能展示太久。

基础追问 3：延迟双删怎么说？

答：延迟双删是为了解决并发读写下旧值回填问题：更新 DB 前后删一次，延迟一小段时间再删一次。但它不是银弹，延迟时间难精确，仍需要 TTL、重试和监控兜底。

你要避免的坑

  • 不要说“更新 DB 同时更新缓存”就结束，双写失败更复杂。
  • 不要把缓存当准源，订单状态应以数据库为准。
  • 不要只背延迟双删，要说删除失败、TTL、版本号。

Case 4 / SQL Optimization

推荐总述

我做过订单查询链路的 SQL 优化，不是简单加索引，而是先从接口耗时和日志定位具体 SQL，再通过 EXPLAIN 看 key、rows、Extra、回表、排序和扫描行数。遇到过几类典型问题：左模糊导致索引失效；WHERE 和 ORDER BY 各有索引时优化器选错执行计划；深分页 OFFSET 触发大量回表。优化时会结合查询条件重构组合索引、必要时使用 hint、把主键加入排序保证稳定，或者通过覆盖索引子查询先分页主键再回表查详情。

案例 A：左模糊索引失效

不要只说“左模糊导致索引失效”。要解释：B+ 树按索引值有序，LIKE 'abc%' 能定位范围，LIKE '%abc%' 无法确定起点，只能扫描大量数据。优化方向是改右模糊、先用高选择性条件缩小范围，或者引入搜索能力。

案例 B：优化器选错索引

WHERE 条件和 ORDER BY 都有索引时，优化器可能为了避免排序选择排序索引，但过滤性很差，实际扫描大量无关行。可以通过组合索引、hint、稳定排序字段来纠正执行计划。

案例 C：深分页大量回表

OFFSET 越大，跳过的行越多。如果直接 select *，被跳过的行也可能发生回表。优化方式是先用覆盖索引子查询拿当前页主键，再回表查少量详情，也可以改成游标翻页。

验证方式

  • EXPLAIN 里 key 是否符合预期。
  • rows 扫描行数是否下降。
  • Extra 是否减少 filesort / temporary。
  • 接口耗时、P95/P99 是否下降。
  • 业务结果是否和优化前一致。
深分页优化模型：
select o.*
from order_table o
join (
    select id
    from order_table
    where tenant_id = ?
    order by create_time desc, id desc
    limit 100000, 20
) t on o.id = t.id
order by o.create_time desc, o.id desc;

基础追问 1：什么是回表？

答：二级索引叶子节点通常存索引列和主键。查询字段不在索引里时，需要根据主键回到聚簇索引查整行，这就是回表。深分页如果跳过大量行还回表，成本会很高。

基础追问 2：组合索引怎么设计？

答：一般把等值过滤列放前面，比如租户、状态；范围和排序列放后面，比如时间；如果 ORDER BY 要稳定，可以加主键作为第二排序字段。目标是同时兼顾过滤和排序。

基础追问 3：Hint 是不是坏味道？

答：Hint 不应该滥用，因为数据分布变化后可能不再适合。但在优化器明显误判、线上急需止血、且查询模式稳定时，可以作为工程手段，同时最好配合索引设计和统计信息分析。

你要避免的坑

  • 不要说“加索引就好了”，要讲执行计划。
  • 不要只看平均耗时，要看 P95/P99 和扫描行数。
  • 不要忽略排序、回表、深分页这些隐藏成本。

案例 5：订单高并发怎么讲具体

“高并发”不能是口号，要拆成创建、查询、回写三个入口分别回答。

Case 5 / Concurrency

推荐回答

订单模块的高并发我会分三类讲。第一是创建侧高并发，主要风险是重复入库和重复下发，靠幂等键、唯一索引、异步任务和下游 requestId 控制。第二是查询侧高并发，比如订单列表和状态轮询，主要风险是慢 SQL 和数据库压力，靠索引优化、分页控制、缓存和热点保护。第三是状态回写侧并发，比如回调、轮询、重试和补偿同时推进同一笔订单，主要风险是状态乱序覆盖，靠状态机、前置状态和版本号 CAS 控制。

创建侧

  • 风险：重复订单、重复任务、重复下发。
  • 方案：幂等键、唯一索引、异步任务、下游 requestId。

查询侧

  • 风险：慢 SQL、连接池压力、热点订单频繁查询。
  • 方案：组合索引、分页限制、缓存、热点互斥回源。

回写侧

  • 风险：成功失败乱序、旧消息覆盖新状态。
  • 方案：状态机、CAS、终态保护、补偿任务。

基础追问：限流和降级呢？

创建入口可以按租户、用户、资源类型做限流；查询入口可以限制深分页和刷新频率；下游异常时可以隔离线程池、熔断或进入排队状态，避免拖垮主链路。

基础知识追问防守卡

这些是面试官很可能顺着订单案例往下追的基础点。

Defense

幂等、事务、锁

幂等是什么？

同一个业务请求执行一次和执行多次，最终业务结果一致。订单创建的幂等不是“不执行”，而是重复请求返回同一笔订单。

唯一索引为什么可靠？

因为并发冲突最终会落到数据库写入约束上，由存储层保证同一 key 只能有一条记录。

事务边界怎么定？

订单主记录、明细、下发任务应在一个本地事务里；外部下游调用不要放在这个事务里。

分布式锁能不能用？

可以作为削峰或保护手段，但不能替代唯一索引。锁失败时，数据库约束仍要兜底。

Defense

缓存、状态机、SQL

缓存一致性怎么收束？

数据库为准，缓存减压。写后删缓存，失败重试，TTL 兜底，必要时版本号防旧值回填。

状态机为什么必要？

它约束合法流转，防止随意 update 导致状态倒退、终态被覆盖、失败重试不可追踪。

EXPLAIN 看什么？

重点看 key、type、rows、Extra。判断是否走对索引、扫描多少行、是否 filesort/temporary、是否回表。

为什么深分页慢？

OFFSET 要跳过大量行。如果查询字段不被索引覆盖，被跳过的数据也可能回表，成本被放大。

我在云服务管理编排平台里重点参与过订单链路和服务下发相关研发。这个订单模块不是简单的增删改查，而是承接上游资源申请，经过网关鉴权、租户和配额校验后，生成订单主记录和资源明细，再异步调用下游服务完成资源下发，最后通过状态回写或轮询更新订单状态。这个链路里我重点关注三类问题：创建侧的幂等和重复入库，通过业务幂等键和数据库唯一索引兜底；状态侧的一致性，通过状态机、版本号或前置状态 CAS 防止并发乱序覆盖；查询侧的性能问题，通过日志、执行计划和索引重构，把大数据量订单查询从几十秒优化到秒级以内。整体来说，我对订单模块的理解是：它是一个状态驱动的业务编排链路，核心不是 CRUD，而是并发、一致性、异步下发和可观测闭环。
