订单模块不是 CRUD，而是状态驱动的业务编排链路。

这个专题围绕云服务管理编排平台中的订单模块展开，重点准备面试官最爱追问的部分：订单主链路、高并发提交、幂等、重复入库、缓存一致性、状态机、异步下发、慢 SQL 和项目表达。

项目背景与职责

先把项目定位讲清楚：你不是只写接口，而是在做云服务资源申请到下发完成的编排链路。

Project Context

一句话项目定位

我参与的是一个云服务管理编排平台，平台负责承接上游资源申请请求，完成网关鉴权、租户校验、订单创建、服务下发、状态回写和查询展示。我的重点工作在订单模块和服务下发链路，处理过订单状态流转、异步下发、查询性能、慢 SQL 和并发场景下的一致性问题。

我的职责

  • 参与订单创建、订单查询、状态推进相关功能研发。
  • 参与服务下发链路问题定位，处理下游调用超时、失败、状态卡住等问题。
  • 排查订单查询慢、索引不合理、并发下性能下降等问题。
  • 通过日志、执行计划和索引调整完成订单查询性能优化。

面试官关注点

  • 订单到底是不是核心链路。
  • 有没有高并发、重复请求和重试场景。
  • 如何避免重复入库和状态乱序。
  • 缓存和数据库如何保持一致。
  • 慢 SQL 是怎么定位和优化的。

订单主链路怎么讲

链路讲清楚后，幂等、缓存、状态机、SQL 优化才都有落点。

Order Flow

从上游请求到最终状态展示

上游请求 / 页面提交
  -> 网关鉴权
  -> 租户、权限、配额校验
  -> 参数校验和业务编排
  -> 生成业务幂等键
  -> 创建订单主记录
  -> 创建订单明细 / 资源参数
  -> 写入异步下发任务
  -> 调用下游服务完成资源下发
  -> 状态轮询 / 回调回写
  -> 查询侧展示订单最终状态

为什么不是简单 CRUD

订单记录只是载体，真正复杂的是订单背后承载了租户校验、资源参数、下游服务调用、异步状态推进和查询展示。一条订单的最终状态依赖多个系统协同，任何一步超时、重试或乱序都可能造成状态不一致。

面试回答起手式

“我做的订单模块主要负责把上游资源申请转成可追踪、可重试、可查询的订单和下发任务。链路从鉴权和租户校验开始，到订单落库、异步下发、状态回写和查询展示结束。”

高并发场景怎么说

不要空泛说“有高并发”，要说清楚并发打在哪个入口，造成什么风险。

Concurrency

订单模块的并发入口

创建侧并发

  • 用户重复点击提交。
  • 上游接口超时后自动重试。
  • 批量资源申请同时创建多笔订单。
  • 风险是重复入库、重复下发、重复扣减配额。

查询侧并发

  • 订单列表高频刷新。
  • 按状态、租户、时间范围筛选。
  • 深分页或复杂排序。
  • 风险是慢 SQL、数据库热点、连接池被打满。

状态回写并发

  • 下游回调和本地轮询同时更新。
  • 重试任务和补偿任务同时推进。
  • 成功、失败消息乱序到达。
  • 风险是状态覆盖、状态倒退和终态不一致。

回答模板

“高并发主要体现在创建、查询和状态回写三个入口。创建侧防重复，查询侧防慢查询，状态侧防乱序覆盖。对应方案是幂等键加唯一索引、查询索引和缓存减压、状态机 CAS 和补偿任务。”

幂等与重复入库

这是订单模块最容易被追问的点：重复点击、超时重试、并发请求到底会不会插两条。

Idempotency

不要只靠前端防抖，也不要只靠先查后插

核心设计

  • 根据租户、业务请求号、资源类型、关键参数摘要生成业务幂等键。
  • 订单表对幂等键建立唯一约束。
  • 并发插入时，只允许一个请求成功。
  • 另一个请求捕获唯一键冲突后查询已有订单并返回。
  • 下游下发也携带同一个请求标识，避免重复执行。

为什么先查后插不够

并发请求 A 和 B 可能同时查不到订单，然后同时插入。如果没有数据库唯一约束兜底，就会产生重复订单。所以应用层校验可以做，但最终要依赖唯一索引保证一致性。

@Transactional
public OrderView createOrder(CreateOrderCmd cmd) {
    String idemKey = buildIdemKey(cmd);
    try {
        orderMapper.insert(OrderDO.pending(idemKey, cmd));
        taskMapper.insert(DispatchTask.orderCreated(idemKey));
    } catch (DuplicateKeyException e) {
        return orderMapper.findByIdemKey(idemKey);
    }
    return orderMapper.findByIdemKey(idemKey);
}

面试句子：创建订单这类强一致入口，我不会只依赖前端防抖或 Redis 锁，而是用业务幂等键加数据库唯一索引兜底。这样即使接口超时重试或多个请求并发到达，也不会重复入库。

状态机与异步下发

订单状态不能随便 update，重点是合法流转、失败重试、乱序防护和补偿。

State Machine

订单状态要能追踪、能重试、能补偿

PENDING       待处理
PROCESSING    下发中
SUCCESS       成功
FAILED        失败
CANCELING     取消中
CANCELED      已取消

典型流转：
PENDING -> PROCESSING -> SUCCESS
PENDING -> PROCESSING -> FAILED
FAILED  -> PROCESSING  // 重试
PROCESSING -> CANCELING -> CANCELED

状态更新

状态更新要带前置状态或版本号，例如只允许 PROCESSING 更新到 SUCCESS，避免失败回写覆盖成功，或者旧消息让状态倒退。

update order
set status = ?, version = version + 1
where order_id = ?
  and status = ?
  and version = ?;

异步下发

  • 订单创建后写入异步任务，避免长事务里直接调用下游。
  • 任务失败按错误类型决定是否重试。
  • 超时不直接判死，进入处理中并由补偿任务确认最终状态。
  • 长时间 PROCESSING 的订单定期扫描，避免状态卡住。

面试句子：订单状态更新不是简单覆盖，而是按状态机推进。并发回写时用前置状态或版本号做 CAS，异步下发失败时通过重试和补偿任务保证最终状态可追踪。

缓存一致性

订单缓存主要服务查询侧，写侧核心是避免旧状态长期留在缓存里。

Cache Consistency

数据库是准源，缓存用于查询减压

读流程

1. 先读缓存；
2. 未命中读数据库；
3. 回填缓存并设置 TTL；
4. 热点订单可加互斥锁防击穿。

写流程

1. 先更新数据库；
2. 事务提交后删除缓存；
3. 删除失败进入重试；
4. TTL 作为最终兜底。

面试句子：订单状态以数据库为准，缓存用于查询减压。写侧我倾向于先更新数据库，提交后删除缓存，删除失败通过重试和 TTL 兜底。对状态类缓存可以带版本号或更新时间，避免旧版本覆盖新版本。

SQL 优化与查询性能

订单查询是面试官很容易追的点，重点讲清定位链路，而不是只说“加索引”。

SQL Performance

从接口慢定位到执行计划

排查链路

接口耗时日志
  -> 定位具体 SQL 和参数
  -> EXPLAIN 看 key / rows / Extra
  -> 判断索引、回表、排序、分页成本
  -> 调整组合索引或改写 SQL
  -> 对比优化前后耗时和扫描行数

你能讲的案例

  • 左模糊匹配导致索引失效。
  • WHERE 和 ORDER BY 各有索引，优化器选错执行计划。
  • OFFSET 深分页触发大量回表，通过覆盖索引子查询和延迟关联优化。
  • 千万级订单数据查询从几十秒优化到秒级以内。

面试句子：我处理 SQL 慢不会直接拍脑袋加索引，而是先从接口日志定位到 SQL，再用 EXPLAIN 看扫描行数、索引命中、回表和排序情况，最后通过组合索引、查询改写、延迟关联等方式优化，并做优化前后对比。
