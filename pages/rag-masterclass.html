1. 一句话与全局地图

RAG 不是“把文档塞进向量库”。RAG 是用检索系统给大模型提供外部证据，让答案更准确、更新、更可追溯。

RAG 解决什么

  • 模型训练知识过期：用外部知识库补实时信息。
  • 企业私有知识：不用把全部内部文档训练进模型。
  • 幻觉问题：要求答案引用检索证据。
  • 成本问题：比微调更轻，更新文档即可更新知识。

RAG 不解决什么

  • 知识库本身是错的，RAG 也会错。
  • 问题需要复杂业务计算，单靠检索不够。
  • 权限、审计、数据治理缺失时，RAG 可能泄露信息。
  • 召回不到关键证据时，模型仍可能编。

面试总括

我会把 RAG 分成离线索引和在线问答两条链路。离线链路负责文档解析、清洗、切块、Embedding、入库；在线链路负责查询改写、混合检索、重排、上下文压缩、提示词约束、答案生成和评估观测。

采集PDF、网页、数据库、工单、代码、Confluence、飞书、Notion。

检索TopK、混合检索、过滤、重排、上下文压缩。
生成带引用回答、拒答、评估、反馈闭环。

3. 标准 RAG 的各种步骤

可以把 RAG 拆成离线索引链路和在线查询链路。面试时先讲这两条链路，再补每一步的坑。

3.1 离线索引链路

  1. 数据接入：从文件、网页、数据库、对象存储、工单系统接入，记录来源、更新时间、作者、权限。
  2. 文档解析：处理 PDF 版面、图片 OCR、表格、代码块、标题层级，尽量保留结构。
  3. 清洗去重：去页眉页脚、目录噪声、重复段落、空白块、乱码，并处理敏感信息。
  4. 切块：按语义、标题、段落、token 长度、滑动窗口、父子块策略切分。
  5. Embedding：把 chunk 转成向量，同时保留原文、元数据、权限标签。
  6. 索引入库：写入向量库、全文索引、图谱或多种索引，并维护版本。

3.2 在线查询链路

  1. 问题理解：识别意图、实体、时间范围、用户权限、是否需要检索。
  2. 查询改写：把口语问题改成更适合检索的查询，可拆成多个子问题。
  3. 召回：向量检索、关键词检索、混合检索、元数据过滤、图检索。
  4. 重排：用 reranker 或 LLM 对候选 chunk 重新排序，剔除弱相关内容。
  5. 上下文构造：去重、合并父子块、按 token 预算压缩，带引用编号。
  6. 生成回答：提示词要求“只基于证据回答，不足则说明不知道”。
  7. 后处理：格式化、引用、敏感信息过滤、事实核查、缓存。
  8. 观测反馈：记录 query、召回文档、答案、用户反馈、延迟、成本。
用户问题
  -> Query Rewrite / Decompose
  -> Hybrid Retrieve: Dense + BM25 + Metadata Filter
  -> Rerank TopN
  -> Context Compression + Citation
  -> LLM Answer
  -> Faithfulness / Recall / Latency Evaluation

4. 数据、切块与索引技术点

Chunking 怎么选

  • 固定长度：简单稳定，适合纯文本，但可能切断语义。
  • 递归切分：按标题、段落、句子逐级切，通用性好。
  • 语义切分：按语义相似度断点切，质量高但成本更高。
  • 滑动窗口：加 overlap 防止上下文断裂，但会增加冗余。
  • 父子块：小块负责召回，大块负责生成上下文。
  • Late Chunking：先用长上下文模型编码，再切块保留跨段语义。

Chunk 大小经验

  • FAQ：100-300 tokens，问题和答案尽量完整。
  • 技术文档：300-800 tokens，按标题层级保留结构。
  • 法律/合同：500-1200 tokens，保留条款编号和上下文。
  • 代码：按函数、类、文件结构切，不要按固定字符粗切。
  • 表格：转成 Markdown 或结构化 JSON，并保留列名。

Embedding 与索引要点

问题做法为什么
同义词召回差选更好的 embedding，增加 query rewrite，混合检索语义向量不一定覆盖业务词表
长文档丢关键上下文父子块、标题路径、摘要索引、多粒度索引小块易召回，大块利于回答
多租户权限元数据过滤、ACL 标签、检索前权限裁剪不能靠 prompt 防泄露
知识更新增量索引、版本号、删除 tombstone、定时重建避免旧知识和新知识同时命中
重复内容hash 去重、相似度去重、canonical source重复块会挤占上下文窗口

5. 检索、重排与上下文压缩

RAG 的上限往往由检索质量决定。模型再强，如果证据错了，答案也会跟着错。

Dense Retrieval

把问题和 chunk 都转成向量，按向量相似度召回。适合同义表达、自然语言问题，但对数字、专有名词、精确匹配可能不稳。

EmbeddingANN

Sparse Retrieval

BM25/关键词检索，适合术语、编号、错误码、产品名、法律条款等精确匹配。

BM25全文索引

Hybrid Retrieval

把 dense 与 sparse 融合，常用 RRF 或加权分数。生产 RAG 通常优先考虑混合检索。

RRFTopK Fusion

重排为什么重要

向量库负责“粗召回”，reranker 负责“精排序”。常见做法是召回 Top 50，再重排取 Top 5-10 放进上下文。它能显著减少“召回了很多但真正证据排在后面”的问题。

retrieve_top_k = 60
rerank_top_k = 8
context_budget = 6000 tokens

候选文档 = dense_top_30 + bm25_top_30
去重后 -> reranker(query, chunk)
按相关性、权限、时效、来源可信度排序
压缩为可引用上下文

上下文压缩

  • Extractive compression：只抽取与问题相关的句子，引用更稳。
  • Abstractive compression：让模型摘要候选证据，省 token 但有摘要失真风险。
  • MMR 去冗余：既考虑相关性，也避免多个 chunk 内容重复。
  • Contextual RAG：给每个 chunk 增加标题路径、来源说明、术语解释，减少孤立片段误读。

6. 生成、引用与防幻觉

好的 RAG Prompt

你是企业知识库助手。
请只依据给定资料回答。
如果资料不足，请说“当前资料不足以确认”，并列出还需要什么信息。
回答必须包含引用编号，如 [1] [2]。
不要使用资料外的猜测补全事实。

答案后处理

  • 检查每个关键结论是否有引用支撑。
  • 引用来源要能回跳到原文、页码或文档段落。
  • 敏感字段脱敏，按用户权限过滤。
  • 对于高风险问题触发拒答或人工审核。
  • 对结构化问题输出 JSON schema，便于前端消费。

常见幻觉来源

来源表现处理
召回错回答看似有引用，但引用不支持结论提升召回、重排、faithfulness 评估
上下文冲突旧文档和新文档同时命中版本控制、时间过滤、来源优先级
问题歧义用户问“这个政策”，系统不知道是哪一个追问澄清或 query disambiguation
Prompt 太松模型把常识和资料混在一起强制证据引用、拒答规则、答案校验

7. RAG 评估与观测

RAG 的评估要拆开看：检索有没有找对，生成有没有忠实，系统有没有稳定、快、省钱。

检索指标

  • Recall@K：正确证据是否在前 K 个。
  • MRR / NDCG：正确证据排得是否靠前。
  • Context Precision：放进上下文的内容是否相关。

生成指标

  • Faithfulness：答案是否被上下文支持。
  • Answer Relevancy：是否回答了问题。
  • Groundedness：关键结论是否可追溯。

工程指标

  • 延迟：P50/P95/P99。
  • 成本：embedding、rerank、generation token。
  • 可用性：错误率、超时率、缓存命中率。

评估工具怎么用

工具定位适合
RAGASRAG 指标评估快速评估 faithfulness、context precision/recall
DeepEvalPython 测试框架把 LLM/RAG 质量做成 CI 测试
TruLens可解释评估分析输入、上下文、输出之间的质量链路
Arize Phoenix开源观测与 tracing查看请求链路、检索结果、评估分数
LangSmithLangChain 生态 tracing/eval调试链、Agent 和数据集回归
W&B / MLflow / Braintrust实验管理与评估平台团队协作、版本对比、线上回归
重点：不要只看 LLM judge 分数。你还需要人工抽样、黄金问答集、困难负样本、权限测试、时间敏感问题测试和线上反馈闭环。

8. 前沿增强型 RAG

增强型 RAG 的本质是补齐朴素 RAG 的短板：检索不准、问题复杂、上下文冲突、证据不足、需要多跳推理。

范式核心思想适合场景代价
Advanced RAG 查询改写、混合检索、重排、压缩、评估闭环 大多数生产知识库 中等复杂度
HyDE 先让模型生成“假设答案/文档”，再用它去检索 用户问题短、抽象、缺少关键词 多一次 LLM 调用，假设可能偏
Multi-Query RAG 把一个问题扩展成多个检索查询 同一问题有多种表达方式 召回成本上升，需要去重
Query Decomposition 把复杂问题拆成子问题分别检索 多跳问答、对比、归因 链路变长，错误会传播
Corrective RAG / CRAG 检索后判断证据质量，不足则重检索或走外部搜索 开放域、资料不完整、可靠性要求高 需要检索质量判别器
Self-RAG 模型自我判断是否需要检索、证据是否支持、是否要修正 希望动态决定检索和回答策略 训练/提示设计复杂，延迟更高
Agentic RAG Agent 规划、调用多个检索工具、评估证据、循环补查 研究型任务、长链路业务助手 成本、延迟、可控性压力最大
GraphRAG 抽取实体关系形成图谱，按社区/关系路径检索与摘要 跨文档关系、组织知识、供应链、投研、案情分析 构图成本高，实体关系抽取质量决定上限
RAPTOR 递归聚类并摘要，形成树状多层索引 长文档、跨章节综合问答 摘要维护成本，更新链路复杂
Multimodal RAG 检索图片、表格、音频、视频帧、版面结构 合同扫描件、产品手册、图表问答、工业巡检 解析、OCR、视觉 embedding 成本更高

怎么决定用哪种增强

先做低成本增强

  1. 改 chunk 与元数据。
  2. 加 BM25 + 向量混合检索。
  3. 加 reranker。
  4. 建黄金集跑评估。
  5. 做查询改写和上下文压缩。

再做复杂增强

  1. 多跳问题多：Query Decomposition。
  2. 关系问题多：GraphRAG。
  3. 资料不足频繁：CRAG。
  4. 任务需要规划：Agentic RAG。
  5. 长文档综合：RAPTOR / 摘要索引。

9. 生产级 RAG 架构

生产 RAG 不是 notebook，而是有权限、版本、观测、缓存、回滚和评估的系统。

                         Context Builder ─> LLM ─> Answer + Citation
                                                   │
                                                   v
                              Trace / Eval / Feedback / Cache

必须考虑的工程问题

权限

检索前做 ACL 过滤，chunk 级别保留租户、部门、角色、文档权限。不要把无权限内容交给模型再靠 prompt 约束。

更新

支持增量重建、删除同步、版本隔离。线上可用蓝绿索引：新索引构建完成后再切流。

缓存

缓存 query rewrite、检索结果、rerank 结果、最终答案。注意权限和知识版本变化会影响缓存有效性。

成本

减少无效 LLM 调用，优先小模型做改写/判断，rerank 控制候选数，按业务价值选择上下文长度。

稳定性

向量库、LLM、reranker 都要有超时、降级和熔断。必要时退化为关键词搜索或只返回相关文档。

安全

防 prompt injection、越权检索、敏感信息泄漏、恶意文档指令。文档内容是数据，不是系统指令。

10. 面试回答模板与追问

Q1：请你讲一下 RAG 的完整流程。
我会分成离线和在线两条链路。离线链路从数据源采集文档，做解析、清洗、切块、Embedding，然后把向量、原文、元数据和权限标签写入索引。在线链路接收用户问题后，先做意图识别和查询改写，再用向量检索、BM25 或混合检索召回候选文档，经过 reranker 排序和上下文压缩，最后把带引用的证据交给大模型生成答案。生成后还会做引用校验、敏感信息过滤、日志追踪和质量评估。
Q2：RAG 和微调有什么区别？
RAG 适合知识更新频繁、需要引用来源、使用企业私有资料的场景，更新知识只需更新索引。微调更适合改变模型风格、格式、任务习惯或领域能力，但不适合频繁注入大量事实知识。实际项目里常见组合是：RAG 管事实，微调或 prompt 管表达和流程。
Q3：如何提升 RAG 准确率？
先定位是检索问题还是生成问题。检索问题从文档解析、chunk、embedding、混合检索、metadata filter、reranker 入手；生成问题从 prompt 约束、引用校验、上下文压缩、拒答策略入手。最后用黄金问答集评估 Recall@K、context precision、faithfulness，并对线上 bad case 做闭环。
Q4：为什么要混合检索？
向量检索擅长语义相似，但对编号、错误码、人名、专有名词、精确短语可能不稳定；BM25 擅长关键词和精确匹配，但不懂同义表达。混合检索把两者结果融合，再加 reranker，可以兼顾召回率和精确度。
Q5：GraphRAG 适合什么？
GraphRAG 适合实体关系密集、需要跨文档综合的问题，比如“某个客户和哪些产品、合同、风险事件有关”。它会抽取实体和关系，形成图谱或社区摘要，再结合向量检索回答。代价是抽取、构图、更新和一致性维护更复杂。
Q6：RAG 如何处理权限？
权限必须在检索前处理。每个 chunk 写入租户、部门、角色、文档 ACL 等元数据，查询时先根据用户身份做过滤，再检索和生成。不能把无权限内容交给 LLM 后再要求它别说，因为 prompt 约束不是安全边界。
Q7：RAG 的常见失败模式有哪些？
文档解析丢结构、chunk 切得太碎或太大、embedding 不适合领域、召回 TopK 太小、没有 rerank、旧知识与新知识冲突、上下文超过 token 预算、prompt 没有拒答、没有评估集、没有线上 tracing。

11. 自测题与记忆卡

RAG 核心检索外部证据，增强模型回答；不是训练模型记住所有文档。
最佳优先级先修数据和检索，再调 prompt；证据不对，生成再漂亮也没用。
最常见增强混合检索 + rerank + query rewrite + context compression。
生产底线权限前置、引用可追溯、评估可回归、链路可观测。
