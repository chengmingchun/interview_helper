HashMap 和 TreeMap 的区别HashMap 基于哈希表，JDK 8 底层是数组、链表和红黑树，平均查询和插入复杂度是 O(1)，但不保证顺序。TreeMap 基于红黑树，查询和插入是 O(logN)，会根据 key 的自然顺序或 Comparator 排序，并支持范围查询。需要快速按 key 查找通常选 HashMap；需要有序遍历、找最近 key 或范围检索时选 TreeMap。
30 秒版

速度 vs 顺序

HashMap 用哈希定位，平均 O(1)、无序；TreeMap 用红黑树，O(logN)、按 key 有序，还支持 floorKey、ceilingKey、subMap。

追问版

不能只说 TreeMap 更慢

TreeMap 的价值是维护顺序与范围关系。只做精确查 key 选 HashMap；要找某时间点之前最近记录，TreeMap 更自然。

HashMap 的 put 流程

按流程说，比死背字段更稳定。

1. 计算 hash对 key.hashCode() 做扰动，让高位也参与桶定位。
2. 定位桶数组长度为 2 的幂，通过 (n - 1) & hash 计算下标。
3. 空桶插入当前位置为空，直接创建 Node。
4. 冲突处理key 相同则覆盖；否则沿链表或红黑树查找。
5. 树化判断链表达到 8 且数组容量至少 64 时才树化，否则优先扩容。
6. 扩容size 超过 threshold，通常扩为 2 倍，节点留在原位或原位 + oldCap。
为什么长度是 2 的幂

位运算定位与扩容拆分

(n - 1) & hash 定位快；翻倍后只需看 hash 新增高位，决定留原位还是移动 oldCap。

为什么树化有两个条件

小数组先扩容更划算

链表长可能只是容量太小。只有节点达到 8 且数组至少 64，才认为值得转红黑树。

线程不安全

更新与复合操作可能竞态

多个线程 put 可能覆盖更新、丢数据；containsKey + put 也不是原子操作。

哈希冲突

不同 key 落到同一个桶

HashMap 用链表或红黑树处理冲突，再通过 equals 判断逻辑上的 key 是否相同。

TreeMap 为什么有序

它不是 HashMap 自动排序版，底层结构完全不同。

底层

红黑树

近似平衡二叉搜索树，通过旋转和变色控制树高，所以查找、插入、删除保持 O(logN)。

排序

Comparable 或 Comparator

key 实现自然顺序，或构造时传比较器。比较结果为 0 时会被视为同一个 key。

范围能力

floor / ceiling / subMap

找不大于某 key 的最近值、找不小于某 key 的最近值，以及区间视图，是 TreeMap 的核心价值。

陷阱

比较器必须稳定

Comparator 不满足传递性会破坏树语义；比较结果和 equals 不一致也会产生反直觉行为。

场景

按时间找最近配置

给定生效时间，使用 floorEntry(time) 找到不晚于它的最近版本。

不适合

只做高频精确查询

没有顺序和范围需求时，HashMap 通常更直接。

ConcurrentHashMap 必须接住的追问

面试官常从 HashMap 顺势追到并发。

问题回答主干常见坑
JDK 8 如何保证线程安全？数组槽位为空时主要用 CAS；桶内冲突时对桶头节点 synchronized；读操作依赖 volatile 可见性。不要回答成 JDK 7 的 Segment 分段锁。
为什么不允许 null？get 返回 null 时无法区分 key 不存在还是 value 就是 null。不要只说设计规定。
putIfAbsent 有什么用？把检查不存在和插入合成原子操作，避免 containsKey 后 put 的竞态。注意返回值语义。
computeIfAbsent 风险？计算函数不应耗时、递归修改同一 Map 或产生复杂副作用。长时间计算可能阻塞同桶操作。
迭代是否强一致？迭代器是弱一致，可反映部分并发更新，不抛 ConcurrentModificationException。弱一致不等于随意错误。
经典追问ConcurrentHashMap 的单次 put 是线程安全的，但 if (!map.containsKey(k)) map.put(k, v) 是两个操作，整体不原子。应使用 putIfAbsent 或 computeIfAbsent。
