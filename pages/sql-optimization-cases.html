SQL Deep Dive / Index · Optimizer · Pagination

把 SQL 优化案例从“我当时这么改”变成“我知道为什么这么改”。

本页根据经验贴口述做匿名化整理：不保留真实表名、字段名、接口名和内部业务代号。三个案例分别对应索引失效、优化器选错执行计划、深分页回表。

先把这三个案例翻译成人话

它们不是三个零散技巧，而是 SQL 优化的三类高频问题。

总览

三类问题其实分别在问三件事

1. 索引为什么没用上

左模糊 LIKE '%xxx' 破坏 B+ 树从左到右有序匹配，索引无法定位起点，只能扫描大量数据。

2. 优化器为什么选错路

WHERE 和 ORDER BY 都有索引时，优化器会估算成本。有时它以为走排序索引更便宜，结果过滤大量无效行。

3. 为什么 OFFSET 越大越慢

深分页要跳过大量行，如果直接查整行，会发生大量回表。优化思路是先用覆盖索引拿到主键，再回表查少量数据。

面试总句

我遇到的 SQL 慢不只是“没加索引”，还包括索引失效、优化器选错索引、深分页回表这三类问题。我会先看执行计划，再判断扫描行数、key、Extra、回表和排序成本。

Diagnostic Flow

从接口慢到优化验证的 5 步

1. 定位 SQL日志、trace、慢查询日志找具体语句。
2. 看 EXPLAIN关注 type、key、rows、Extra。
3. 判断成本索引失效、扫描过多、排序、回表。
4. 改写方案改条件、组合索引、覆盖索引、延迟关联。
5. 验证回归对比耗时、扫描行数、结果正确性。

当前重点：先定位具体 SQL，别凭感觉优化接口。一个接口慢可能由多个 SQL、远程调用或序列化造成。

案例 1：SQL 超时，左模糊匹配导致索引失效

关键词：B+ 树、最左前缀、LIKE、扫描行数。

Index Invalid

现象 → 原因 → 优化

现象

  • 订单或资源列表查询超时。
  • 查询条件里有名称、标识、描述类字段的模糊搜索。
  • 字段本身有索引，但执行计划扫描行数仍然很大。

根因

B+ 树索引要求从左到右匹配。LIKE 'abc%' 能定位到 abc 开头的范围；LIKE '%abc' 或 LIKE '%abc%' 不知道从哪里开始，只能扫描。

-- 不友好：左模糊，普通 B+ 树索引很难利用
where name like '%keyword%'

-- 可优化方向 1：改成右模糊，保留索引范围扫描能力
where name like 'keyword%'

-- 可优化方向 2：如果必须任意位置匹配，考虑搜索引擎、倒排索引、冗余字段或业务改造
-- 可优化方向 3：先用租户、状态、时间等高选择性条件缩小范围，再做模糊过滤

面试别只说“左模糊索引失效”。要补一句：因为 B+ 树按索引值有序，左侧通配符让数据库无法确定扫描起点。

案例 2：SQL 未下发，WHERE 和 ORDER BY 各持索引导致优化器误判

关键词：执行计划、成本估算、索引选择、Hint、排序索引。

Optimizer Misjudge

这个案例你原话里的“解释器误判”，更准确叫优化器成本估算误判

现象

  • 业务上表现为 SQL 没有按预期下发或执行非常慢。
  • 排查相关会话/明细表数量正常，说明不是数据没生成。
  • 执行计划发现数据库没有选择过滤性更好的索引。

根因

WHERE 条件能走一个索引，ORDER BY 又能走另一个索引。优化器可能为了避免额外排序，选择了排序索引，但这个索引过滤性差，导致扫描大量无关数据。

-- 抽象问题模型：
select *
from order_table
where tenant_id = ? and status = ?
order by create_time desc
limit 20;

-- 可能出现两条路：
-- A. 走 where 条件索引：过滤快，但可能需要 filesort
-- B. 走 order by 索引：天然有序，但可能过滤大量无关行

-- 优化方向 1：建立更匹配查询的组合索引
(tenant_id, status, create_time)

-- 优化方向 2：必要时用 hint 指定索引，避免优化器误判
force index(idx_tenant_status_time)

-- 优化方向 3：把主键加入 order by，保证排序稳定，也帮助索引设计
order by create_time desc, id desc

你说的“增加 hint 或者主键进入 order by”，可以理解成：要么明确告诉优化器走哪个索引，要么把排序条件设计得更稳定、更贴合组合索引。

案例 3：OFFSET 触发大量回表，先分页主键再回表

关键词：深分页、覆盖索引、回表、延迟关联。

Deep Pagination

OFFSET 慢不是因为 limit，而是因为跳过的那些行也要付成本

现象

  • 第一页很快，越往后翻越慢。
  • limit 100000, 20 这类深分页耗时明显。
  • 查询字段很多，索引无法覆盖，需要大量回表。

根因

数据库需要先找到并跳过 offset 行。如果每一行都查整行，就会对大量被丢弃的数据做回表，成本很高。

-- 原始写法：深分页 + 查整行，容易大量回表
select *
from order_table
where tenant_id = ?
order by create_time desc
limit 100000, 20;

-- 优化写法：先用覆盖索引拿到这一页的主键
select o.*
from order_table o
join (
    select id
    from order_table
    where tenant_id = ?
    order by create_time desc, id desc
    limit 100000, 20
) t on o.id = t.id
order by o.create_time desc, o.id desc;

-- 更进一步：游标翻页，避免大 offset
where (create_time, id) < (?, ?)
order by create_time desc, id desc
limit 20;

你说的“右链接查询”，面试里建议说成“延迟关联 / 子查询先分页主键，再按主键回表查详情”。这样更通用，也更容易被理解。

Interview Script

90 秒版本

我做过几类订单查询 SQL 优化。第一类是索引失效，比如模糊查询用了左通配，B+ 树无法定位扫描起点，导致即使字段有索引也会扫很多行；这类要么改成右模糊，要么先用租户、状态、时间等高选择性条件缩小范围，必要时引入搜索能力。第二类是优化器选错索引，WHERE 和 ORDER BY 都有索引时，优化器可能为了避免排序走了排序索引，但过滤性很差，实际扫描更多；我会通过 EXPLAIN 看 key、rows、Extra，必要时改组合索引、加 hint，或者把主键加入排序保证稳定。第三类是深分页回表，OFFSET 很大时如果直接查整行，会对大量被跳过的数据回表；优化方式是先用覆盖索引子查询拿到当前页主键，再关联主表查详情，或者改成游标翻页。我的习惯是先定位具体 SQL，再看执行计划，最后做优化前后耗时和扫描行数对比。

口述框架：
现象：接口超时 / SQL 未按预期执行 / 深分页变慢
排查：日志定位 SQL -> EXPLAIN -> key / rows / Extra
原因：索引失效 / 优化器误判 / 大量回表
方案：改查询模式 / 组合索引或 hint / 延迟关联
验证：执行计划变化 + 耗时下降 + 回归业务结果
