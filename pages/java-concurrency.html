2. 创建线程：四种写法与推荐顺序

面试里要知道各种写法，但工程里通常不要手动频繁 `new Thread()`，而是把任务交给线程池。

2.1 继承 Thread：能用，但耦合高

class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " running");
    }
}

public class Demo {
    public static void main(String[] args) {
        new MyThread().start(); // 注意：调用 start() 才是启动新线程
        // new MyThread().run(); // 只是普通方法调用，不会开启新线程
    }
}

2.2 实现 Runnable：任务和线程分离

public class RunnableDemo {
    public static void main(String[] args) {
        Runnable task = () -> {
            String name = Thread.currentThread().getName();
            System.out.println(name + " handles order");
        };

        Thread t1 = new Thread(task, "worker-1");
        Thread t2 = new Thread(task, "worker-2");
        t1.start();
        t2.start();
    }
}

2.3 Callable + Future：有返回值和异常

import java.util.concurrent.*;

public class CallableDemo {
    public static void main(String[] args) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(2);

        Callable<Integer> task = () -> {
            TimeUnit.MILLISECONDS.sleep(200);
            return 42;
        };

        Future<Integer> future = pool.submit(task);
        Integer result = future.get(1, TimeUnit.SECONDS);
        System.out.println("result = " + result);

        pool.shutdown();
    }
}

2.4 线程池：生产更推荐

import java.util.concurrent.*;

public class PoolSubmitDemo {
    public static void main(String[] args) {
        ExecutorService pool = new ThreadPoolExecutor(
            2,
            4,
            60,
            TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(100),
            r -> new Thread(r, "biz-worker-" + System.nanoTime()),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );

        for (int i = 0; i < 10; i++) {
            int taskId = i;
            pool.execute(() -> System.out.println("task " + taskId));
        }

        pool.shutdown();
    }
}
面试说法：创建线程有继承 Thread、实现 Runnable、实现 Callable、提交到线程池等方式。实际项目里我优先用线程池，因为它能复用线程、限制并发、统一命名、处理队列和拒绝策略。

3. 线程状态与生命周期

状态含义常见触发
NEW创建了线程对象，还没 `start``new Thread()`
RUNNABLE可运行，可能正在 CPU 上跑，也可能等待 CPU 时间片`start()` 后
BLOCKED等待进入 synchronized 监视器锁竞争对象锁
WAITING无限期等待其他线程唤醒`wait()`、`join()`、`park()`
TIMED_WAITING带超时时间的等待`sleep`、`wait(timeout)`、`get(timeout)`
TERMINATED线程执行结束`run` 方法结束或抛出未捕获异常

join：等待另一个线程结束

public class JoinDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread downloader = new Thread(() -> {
            try {
                Thread.sleep(300);
                System.out.println("file downloaded");
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        downloader.start();
        downloader.join(); // main 等 downloader 结束
        System.out.println("start parsing file");
    }
}

中断：不是强杀线程，而是协作取消信号

public class InterruptDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                try {
                    Thread.sleep(100);
                    System.out.println("working...");
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt(); // 恢复中断标记
                }
            }
            System.out.println("exit safely");
        });

        worker.start();
        Thread.sleep(350);
        worker.interrupt();
    }
}

4. JMM 与 volatile：可见性、原子性、有序性

Java 内存模型要解决的是：多个线程读写共享变量时，一个线程的修改什么时候对另一个线程可见，以及指令重排会不会破坏并发语义。

可见性

线程 A 修改共享变量，线程 B 不一定立刻看到。`volatile`、锁释放/获取、线程启动/结束等会建立 happens-before。

原子性

`count++` 不是原子操作，它包含读、加、写三步。要用锁或原子类保证复合操作安全。

有序性

编译器和 CPU 可能重排指令，只要单线程语义不变。并发下要靠内存屏障和同步规则限制重排。

volatile 适合做停止标记

public class VolatileStopDemo {
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (running) {
                // do work
            }
            System.out.println("stopped");
        });

        worker.start();
        Thread.sleep(100);
        running = false; // 对 worker 可见
    }
}

volatile 不保证 count++ 原子性

public class VolatileNotAtomic {
    private static volatile int count = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[10];
        for (int i = 0; i < threads.length; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < 10_000; j++) {
                    count++; // 读 -> 加 -> 写，多个线程会互相覆盖
                }
            });
            threads[i].start();
        }

        for (Thread t : threads) {
            t.join();
        }
        System.out.println(count); // 通常小于 100000
    }
}

双重检查锁定：为什么 instance 要 volatile

public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

`new Singleton()` 可能被拆成分配内存、初始化对象、引用赋值。没有 `volatile` 时，另一个线程可能看到“引用不为空但对象尚未初始化完成”的状态。

5. synchronized：内置锁、可重入、对象锁

`synchronized` 同时解决互斥和可见性：进入同步块要获取锁，退出同步块会释放锁并刷新修改。

5.1 线程不安全反例

public class UnsafeCounter {
    private int count = 0;

    public void increment() {
        count++;
    }

    public int get() {
        return count;
    }
}

5.2 synchronized 修复

public class SafeCounter {
    private int count = 0;

    public synchronized void increment() {
        count++;
    }

    public synchronized int get() {
        return count;
    }
}

5.3 锁对象不同，保护不了同一份数据

public class WrongLockDemo {
    private int count = 0;

    public void increment() {
        Object lock = new Object(); // 每次都是新锁，等于没锁
        synchronized (lock) {
            count++;
        }
    }
}

5.4 正确锁住共享对象

public class RightLockDemo {
    private final Object lock = new Object();
    private int count = 0;

    public void increment() {
        synchronized (lock) {
            count++;
        }
    }
}

5.5 wait/notify：必须在 synchronized 内使用

import java.util.LinkedList;
import java.util.Queue;

public class WaitNotifyQueue<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    public WaitNotifyQueue(int capacity) {
        this.capacity = capacity;
    }

    public synchronized void put(T item) throws InterruptedException {
        while (queue.size() == capacity) {
            wait(); // 用 while 防止虚假唤醒
        }
        queue.offer(item);
        notifyAll();
    }

    public synchronized T take() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();
        }
        T item = queue.poll();
        notifyAll();
        return item;
    }
}
记忆点：`wait` 会释放锁并进入等待队列；`sleep` 不释放锁；`notify` 只唤醒一个等待线程；`notifyAll` 唤醒全部等待线程，由它们重新竞争锁。

6. Lock、Condition、读写锁

`Lock` 比 `synchronized` 更灵活：可中断、可超时、可公平、可绑定多个条件队列。但要记得在 `finally` 中释放锁。

6.1 ReentrantLock 基本用法

import java.util.concurrent.locks.ReentrantLock;

public class ReentrantLockCounter {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
}

6.2 tryLock：避免无限等待

import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.ReentrantLock;

public class TryLockDemo {
    private final ReentrantLock lock = new ReentrantLock();

    public boolean update() throws InterruptedException {
        if (lock.tryLock(200, TimeUnit.MILLISECONDS)) {
            try {
                // update shared state
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // 降级或稍后重试
    }
}

6.3 Condition：两个条件队列实现阻塞队列

import java.util.ArrayDeque;
import java.util.Queue;
import java.util.concurrent.locks.*;

public class ConditionQueue<T> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Queue<T> queue = new ArrayDeque<>();
    private final int capacity;

    public ConditionQueue(int capacity) {
        this.capacity = capacity;
    }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == capacity) {
                notFull.await();
            }
            queue.offer(item);
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                notEmpty.await();
            }
            T item = queue.poll();
            notFull.signal();
            return item;
        } finally {
            lock.unlock();
        }
    }
}

6.4 ReentrantReadWriteLock：读多写少

import java.util.*;
import java.util.concurrent.locks.*;

public class CacheWithReadWriteLock {
    private final Map<String, String> cache = new HashMap<>();
    private final ReentrantReadWriteLock rw = new ReentrantReadWriteLock();

    public String get(String key) {
        rw.readLock().lock();
        try {
            return cache.get(key);
        } finally {
            rw.readLock().unlock();
        }
    }

    public void put(String key, String value) {
        rw.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            rw.writeLock().unlock();
        }
    }
}

6.5 StampedLock：乐观读

import java.util.concurrent.locks.StampedLock;

public class Point {
    private double x;
    private double y;
    private final StampedLock lock = new StampedLock();

    public void move(double deltaX, double deltaY) {
        long stamp = lock.writeLock();
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            lock.unlockWrite(stamp);
        }
    }

    public double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead();
        double currentX = x;
        double currentY = y;
        if (!lock.validate(stamp)) {
            stamp = lock.readLock();
            try {
                currentX = x;
                currentY = y;
            } finally {
                lock.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }
}

7. 原子类与 CAS：无锁并发的常见入口

原子类基于 CAS 思想：比较当前值是否还是预期值，是则更新，否则失败重试。适合简单计数、状态切换、引用更新。

7.1 AtomicInteger 修复 count++

import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public int increment() {
        return count.incrementAndGet();
    }

    public int get() {
        return count.get();
    }
}

7.2 compareAndSet 做状态流转

import java.util.concurrent.atomic.AtomicReference;

public class OrderStateMachine {
    enum State { CREATED, PAID, CANCELED }

    private final AtomicReference<State> state =
        new AtomicReference<>(State.CREATED);

    public boolean pay() {
        return state.compareAndSet(State.CREATED, State.PAID);
    }

    public boolean cancel() {
        return state.compareAndSet(State.CREATED, State.CANCELED);
    }
}

7.3 LongAdder：高并发计数更适合

import java.util.concurrent.atomic.LongAdder;

public class QpsCounter {
    private final LongAdder requests = new LongAdder();

    public void record() {
        requests.increment();
    }

    public long snapshot() {
        return requests.sum();
    }
}

`AtomicLong` 在高并发下所有线程竞争一个变量；`LongAdder` 分散到多个槽位，最后汇总，适合统计类计数，不适合要求每次读取都严格最新的场景。

8. 并发容器：不要拿 HashMap 硬扛并发

容器适合场景注意点
ConcurrentHashMap高并发读写 Map单个操作线程安全，复合逻辑仍要额外控制
CopyOnWriteArrayList读多写少，如监听器列表写时复制成本高，不适合频繁写
BlockingQueue生产者消费者、线程池任务队列要选择容量，避免无界队列导致内存风险
ConcurrentLinkedQueue非阻塞队列没有阻塞等待语义

8.1 ConcurrentHashMap 的 computeIfAbsent

import java.util.concurrent.*;

public class CacheDemo {
    private final ConcurrentHashMap<String, String> cache = new ConcurrentHashMap<>();

    public String getValue(String key) {
        return cache.computeIfAbsent(key, this::loadFromDb);
    }

    private String loadFromDb(String key) {
        return "value-of-" + key;
    }
}

8.2 复合操作仍可能需要锁

import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicBoolean;

public class LimitRegistry {
    private final ConcurrentHashMap<String, Integer> used = new ConcurrentHashMap<>();
    private final int limit = 10;

    public boolean tryAcquire(String userId) {
        AtomicBoolean acquired = new AtomicBoolean(false);
        used.compute(userId, (k, oldValue) -> {
            int current = oldValue == null ? 0 : oldValue;
            if (current >= limit) {
                return current;
            }
            acquired.set(true);
            return current + 1;
        });
        return acquired.get();
    }
}

8.3 BlockingQueue：生产者消费者

import java.util.concurrent.*;

public class BlockingQueueDemo {
    public static void main(String[] args) {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);

        Thread producer = new Thread(() -> {
            try {
                queue.put("order-1");
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        Thread consumer = new Thread(() -> {
            try {
                String order = queue.take();
                System.out.println("handle " + order);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
    }
}

9. 线程池：高频面试点

线程池的核心价值是复用线程、控制并发、隔离资源、统一监控。生产上要避免直接用 `Executors` 的无界队列风险。

9.1 七个核心参数

参数含义面试解释
corePoolSize核心线程数长期保留的线程数量
maximumPoolSize最大线程数队列满后最多扩到多少线程
keepAliveTime空闲存活时间非核心线程空闲多久回收
unit时间单位配合 keepAliveTime
workQueue任务队列常见 ArrayBlockingQueue、LinkedBlockingQueue、SynchronousQueue
threadFactory线程工厂命名线程、设置异常处理器
handler拒绝策略队列满且线程达到最大值时触发

9.2 线程池执行流程

  1. 当前线程数小于核心线程数：直接创建核心线程执行任务。
  2. 核心线程已满：尝试放入任务队列。
  3. 队列也满：尝试创建非核心线程，直到最大线程数。
  4. 最大线程数也满：执行拒绝策略。

9.3 推荐写法：显式创建 ThreadPoolExecutor

import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class ThreadPoolFactory {
    public static ExecutorService newOrderPool() {
        AtomicInteger seq = new AtomicInteger();

        ThreadFactory factory = r -> {
            Thread t = new Thread(r);
            t.setName("order-pool-" + seq.incrementAndGet());
            t.setUncaughtExceptionHandler((thread, ex) ->
                System.err.println(thread.getName() + " failed: " + ex.getMessage()));
            return t;
        };

        return new ThreadPoolExecutor(
            8,
            16,
            60,
            TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(1000),
            factory,
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }
}

9.4 拒绝策略怎么选

策略行为适合
AbortPolicy直接抛异常希望快速暴露问题
CallerRunsPolicy由提交任务的线程执行削峰，给上游施加反压
DiscardPolicy静默丢弃极少直接使用，容易丢任务无感知
DiscardOldestPolicy丢掉队列最老任务只关心最新任务的场景

9.5 优雅关闭

import java.util.concurrent.*;

public class ShutdownDemo {
    public static void shutdownGracefully(ExecutorService pool) {
        pool.shutdown();
        try {
            if (!pool.awaitTermination(10, TimeUnit.SECONDS)) {
                pool.shutdownNow();
            }
        } catch (InterruptedException e) {
            pool.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
}

10. Future、CompletableFuture 与异步编排

`Future` 能拿结果，但组合能力弱；`CompletableFuture` 更适合多个异步任务编排、异常处理和超时控制。

10.1 Future 的局限

Future<String> userFuture = pool.submit(() -> queryUser(userId));
Future<String> orderFuture = pool.submit(() -> queryOrders(userId));

// get 会阻塞，组合逻辑需要手写
String user = userFuture.get(500, TimeUnit.MILLISECONDS);
String orders = orderFuture.get(500, TimeUnit.MILLISECONDS);

10.2 CompletableFuture 并行查询

import java.util.concurrent.*;

public class CompletableFutureDemo {
    private final Executor executor = Executors.newFixedThreadPool(8);

    public String buildProfile(String userId) {
        CompletableFuture<String> userFuture =
            CompletableFuture.supplyAsync(() -> queryUser(userId), executor);

        CompletableFuture<String> orderFuture =
            CompletableFuture.supplyAsync(() -> queryOrders(userId), executor);

        return userFuture.thenCombine(orderFuture, (user, orders) ->
            user + "\n" + orders
        ).orTimeout(800, TimeUnit.MILLISECONDS)
         .exceptionally(ex -> "fallback profile")
         .join();
    }

    private String queryUser(String userId) { return "user:" + userId; }
    private String queryOrders(String userId) { return "orders:[]"; }
}

10.3 allOf：等待多个任务完成

CompletableFuture<String> a = CompletableFuture.supplyAsync(() -> "A");
CompletableFuture<String> b = CompletableFuture.supplyAsync(() -> "B");
CompletableFuture<String> c = CompletableFuture.supplyAsync(() -> "C");

CompletableFuture<Void> all = CompletableFuture.allOf(a, b, c);
all.join();

System.out.println(a.join() + b.join() + c.join());

10.4 异常处理区别

CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> {
        if (true) throw new RuntimeException("remote failed");
        return "ok";
    })
    .exceptionally(ex -> "fallback"); // 只处理异常，返回兜底值

12. ThreadLocal：线程隔离变量与内存泄漏

`ThreadLocal` 给每个线程一份独立变量，常用于用户上下文、traceId、日期格式等。但在线程池中必须 `remove`，否则线程复用会污染上下文。

12.1 存 traceId

public class TraceContext {
    private static final ThreadLocal<String> TRACE_ID = new ThreadLocal<>();

    public static void set(String traceId) {
        TRACE_ID.set(traceId);
    }

    public static String get() {
        return TRACE_ID.get();
    }

    public static void clear() {
        TRACE_ID.remove();
    }
}

12.2 在线程池里必须清理

public void handleRequest(String traceId) {
    try {
        TraceContext.set(traceId);
        // business logic
    } finally {
        TraceContext.clear(); // 防止下一个请求复用线程时读到旧值
    }
}

12.3 为什么会内存泄漏

`ThreadLocalMap` 的 key 是弱引用，但 value 是强引用。在线程池里线程长期存活，如果不 `remove`，value 可能一直挂在线程上，造成内存泄漏或上下文串号。

13. Java 21 虚拟线程：适合 I/O 密集任务

虚拟线程是轻量级线程，适合大量阻塞 I/O 请求。它不是让 CPU 计算变快，也不是替代所有线程池隔离。

13.1 每个请求一个虚拟线程

import java.util.concurrent.*;

public class VirtualThreadDemo {
    public static void main(String[] args) throws Exception {
        try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
            Future<String> future = executor.submit(() -> {
                Thread.sleep(100);
                return Thread.currentThread().toString();
            });
            System.out.println(future.get());
        }
    }
}

public class DeadLockDemo {

  • 用线程栈和 `jstack` 查找 deadlock。

15. 高频面试回答模板

Q1：说说 synchronized 和 ReentrantLock 的区别。
`synchronized` 是 JVM 内置锁，语法简单，异常退出会自动释放锁，适合大多数互斥场景。`ReentrantLock` 是 JUC 提供的显式锁，必须手动 unlock，但支持公平锁、可中断等待、超时获取锁、多个 Condition 条件队列等高级能力。普通互斥我优先用 synchronized，需要超时、可中断或多条件队列时用 ReentrantLock。
Q2：volatile 能保证线程安全吗？
不能笼统说能。`volatile` 保证可见性和一定有序性，不保证复合操作原子性。比如停止标记适合 volatile；但 `count++` 不适合，因为它包含读、加、写三步，要用 synchronized、Lock 或 AtomicInteger。
Q3：线程池的执行流程是什么？
提交任务后，如果当前线程数小于核心线程数，就创建核心线程执行；如果核心线程满了，就尝试入队；如果队列满了，就创建非核心线程直到最大线程数；如果最大线程也满，就执行拒绝策略。这个流程也解释了为什么无界队列会让 maximumPoolSize 失效。
Q4：为什么不推荐 Executors.newFixedThreadPool？
它默认使用无界 LinkedBlockingQueue，如果任务生产速度大于消费速度，队列可能无限堆积导致 OOM。生产中更推荐显式创建 ThreadPoolExecutor，设置有界队列、线程命名、拒绝策略、监控指标和优雅关闭。
Q5：ThreadLocal 为什么会内存泄漏？
ThreadLocalMap 的 key 是弱引用，但 value 是强引用。在线程池场景中线程长期存活，如果任务结束不 remove，value 可能一直被线程持有，导致内存泄漏或请求上下文串号。所以一般在 finally 中调用 remove。
Q6：CountDownLatch 和 CyclicBarrier 区别？
CountDownLatch 是一个或多个线程等待其他线程完成，计数减到 0 后不能重置；CyclicBarrier 是一组线程互相等待到齐，可以循环使用，还可以设置 barrier action。前者像“发令枪等待准备完成”，后者像“多个人到集合点再一起走”。
Q7：Java 21 虚拟线程解决什么问题？
虚拟线程解决的是大量阻塞 I/O 场景下平台线程成本高的问题，让同步写法也能承载大量并发请求。它不提升 CPU 密集计算的速度，也不取消资源限流的必要，数据库连接、下游 QPS、锁竞争仍然要控制。
