Agent Engineering / Tool Use · MCP · RAG · Memory · Eval · Guardrails

严格面试官不会问“你用没用 AI”，而会问“你到底做没做出可控的 Agent 工程闭环”。

这页用来把 AI 辅助研发项目讲成 Agent 工程化项目：Agent 怎么规划、怎么拿上下文、怎么选工具、怎么调用工具、怎么观察结果、怎么纠错、怎么评估、怎么迁移，以及为什么现在仍然有瓶颈。

项目定位：不是 AI 写代码，而是 Agent 化研发助手

开场要先把项目从“我会用 ChatGPT”拉到“我做过研发流程的 Agent 化改造”。

Project Positioning

推荐开场

这个项目是我近半年参与的 AI 辅助研发与智能交付探索。我不是把它理解成“让大模型直接写代码”，而是把研发流程拆成需求理解、上下文检索、任务规划、工具调用、代码生成、测试验证、人工 Review 和转测交付几个可验证环节。Agent 在其中负责规划、选择工具、调用工具、读取观察结果并迭代；人负责需求判断、风险控制和最终验收。试点里我们完成了从 0 到 1 的穿刺验证，累计生成约 3k 行代码，并提前 2 天转测。

要强调的核心

  • 不是单次 Prompt，而是多步骤 Agent loop。
  • 不是只生成代码，而是围绕研发流程输出中间产物。
  • 不是完全自动驾驶，而是人机协同、可审查、可回滚。
  • 不是一个工具，而是 RAG、Tool、MCP、Skill、SubAgent、测试评估组合。

严格面试官想确认什么

  • 你是否分得清 Agent、Workflow、Function Calling 和 ReAct。
  • 你是否知道工具调用为什么会失败、怎么兜底。
  • 你是否有评估指标，而不是只说“体感提效”。
  • 你是否知道 Agent 的边界和安全风险。

严格面试官会怎么问

这部分就是防守地图。你要提前知道对方会把你从“项目描述”追到“工程细节”。

Interviewer Pressure Test

高频追问路线

第一层：是不是套壳？

  • 你这个和普通聊天机器人有什么区别？
  • 为什么叫 Agent，不叫 Workflow？
  • Agent 自主决策体现在哪里？
  • 哪些环节是确定流程，哪些环节由模型决策？

第二层：工具怎么用？

  • 工具列表怎么召回？
  • 工具太多会怎样？
  • 模型选错工具怎么办？
  • MCP/Skill 的输入输出怎么约束？

第三层：怎么证明有效？

  • 生成 3k 行代码有什么意义？
  • 如何评估质量和提效？
  • 怎么观测 Agent 每一步？
  • 线上或真实交付中最大的瓶颈是什么？

回答策略：不要抢着讲概念。先把项目闭环讲清楚，再承认边界。严格面试官其实更相信“我知道哪些地方不能自动化”，而不是“AI 什么都能做”。

Agent 架构：你要能画出组件

Agent 不是一个 LLM，而是 LLM 外围的一套状态、工具、记忆、规划、执行和观测机制。

Agent Architecture

我会这样拆

用户目标 / 研发任务
  -> Orchestrator 调度器
  -> Context Builder 上下文构造
      -> RAG 检索项目知识
      -> Memory 读取会话/偏好/历史决策
  -> Planner 任务规划
  -> Tool Router 工具选择
      -> Skill / MCP / 脚本 / 代码仓库 / Wiki / CI
  -> Executor 执行工具或生成代码
  -> Observer 读取工具结果和日志
  -> Critic / Reviewer 自检与人工 Review
  -> Final Output 输出代码、文档、测试、转测说明

每个组件的作用

  • Orchestrator：控制流程，不让模型无边界乱跑。
  • Context Builder：把需求、代码、文档、历史决策拼成可用上下文。
  • Planner：把目标拆成步骤，决定先做什么后做什么。
  • Tool Router：选择工具，并构造结构化参数。
  • Observer：把工具返回转成模型可理解的观察结果。
  • Reviewer：检查输出是否满足规格、测试和风险约束。

面试回答重点

我不会把 Agent 等同于“大模型接口”。大模型只是推理和生成核心，真正工程化的是外层的上下文管理、工具调用、状态跟踪、错误处理、评估和人工确认机制。

Agent 研发闭环动画推演

把“AI 辅助研发”讲成一条可观测、可验证、可人工接管的工程链路。

Animated Agent Flow

一次需求从输入到转测怎么流动

1. 需求输入用户目标、业务背景、验收标准。
2. 上下文构造RAG 召回代码、文档、历史决策。
3. 任务规划拆步骤、定边界、生成待确认问题。
4. 工具调用MCP/Skill/脚本/仓库检索。
5. 代码与测试生成 diff、单测、Mock、迁移说明。
6. Review 评估自检、人工 Review、测试结果。
7. 转测交付风险清单、回滚方案、交付说明。

当前重点：需求输入不是一句话 Prompt，要提取目标、边界、角色、验收标准和未知问题。

Agent Loop：每一轮到底发生什么

Agent 热门问题里最容易被追：循环如何终止、如何避免无限调用工具、如何处理失败。

Agent Loop

Observe → Think/Plan → Act → Observe → Verify

while not done and step < max_steps:
    observe: 读取用户目标、上下文、历史状态、工具结果
    plan:    选择下一步动作，判断是否需要工具
    act:     调用工具 / 生成代码 / 请求澄清 / 产出文档
    observe: 解析工具返回、错误、日志、diff、测试结果
    verify:  检查是否满足验收标准，必要时重试或降级
return final_answer / diff / test_report / handoff_note

为什么要有循环

研发任务不是一次生成就结束。模型可能需要先检索代码，再生成方案，再调用工具，再根据测试结果修正。Agent loop 让它可以根据观察结果迭代，而不是一次性输出。

怎么防止循环失控

  • 限制最大步数和最大工具调用次数。
  • 工具调用必须有结构化参数校验。
  • 连续失败要停止并输出失败原因。
  • 写操作、部署操作需要人工确认。
  • 每轮保存 trace，便于复盘。

严格面试官追问：什么时候不需要 Agent loop？

如果任务是固定流程、规则明确、输入输出稳定，比如固定格式转换或固定流水线执行，用 Workflow 更稳定、更便宜。Agent loop 适合目标明确但路径不完全确定、需要动态选择工具或根据观察结果调整步骤的任务。

每个环节做了什么、输出什么、为什么这样输出

严格面试官会问“你说流程，那每一步的产物是什么？”

Step 1

需求理解与澄清

做了什么

输入原始需求、会议记录、Wiki、历史缺陷和业务上下文，让 Agent 抽取业务目标、角色、边界条件、影响模块和验收标准。

输出什么

需求摘要、待确认问题、影响范围、风险点、验收标准草案。

为什么这样输出

因为研发里最大的返工不是代码慢，而是需求理解错。先让 Agent 输出澄清问题和验收标准，可以把模糊需求变成可开发任务。

Step 2

规格生成 / SDD

做了什么

让 Agent 把需求转成规格：接口变化、状态变化、数据结构、异常分支、兼容性、灰度和回滚要求。

输出什么

OpenSpec / 规格说明、接口草案、状态流转、数据模型、异常处理清单、验收用例。

为什么这样输出

规格是代码生成的边界。没有规格就生成代码，模型容易自己补业务假设；有规格后，代码、测试、Review 都能围绕同一份约束展开。

Step 3

上下文构造 / RAG

做了什么

从 Wiki、历史方案、接口文档、代码片段、会话总结中召回相关上下文，并对上下文做去重、排序和摘要。

输出什么

相关文档片段、关联模块、历史实现参考、接口约束、风险提示。

为什么这样输出

Agent 没有项目上下文就容易生成“通用正确、项目错误”的方案。RAG 不是替代模型推理，而是把项目事实放到模型面前。

Step 4

任务规划 / SubAgent 分工

做了什么

把复杂研发任务拆成需求分析、方案设计、代码实现、测试验证、Review 审查等角色，每个角色只负责一种产物。

输出什么

任务拆解表、模块修改清单、执行顺序、风险清单、测试计划。

为什么这样输出

一个 Agent 一次性做完整复杂任务容易遗漏。角色化后目标更窄，产物更稳定，也更适合人工逐段审查。

Step 5

工具调用 / MCP / Skill

做了什么

把知识检索、代码搜索、脚本执行、热部署、会话总结、测试执行等高频操作封装为受控工具，让 Agent 通过结构化参数调用。

输出什么

工具调用参数、执行结果、错误信息、日志摘要、下一步建议。

为什么这样输出

模型擅长理解和生成，不擅长可靠执行。工具化把“想做什么”和“怎么执行”分开，降低幻觉执行和手工操作成本。

Step 6

代码生成、测试与转测

做了什么

基于规格和上下文生成代码 diff、配置变更、单测、Mock 数据和变更说明，再通过人工 Review、测试执行和转测清单收口。

输出什么

代码 diff、文件变更清单、单测结果、Review 清单、转测说明、风险说明。

为什么这样输出

代码只是交付物之一。严格面试官会更关注“能不能验收”。所以必须把测试结果、风险说明和转测说明一起输出。

Agent / Workflow / Function Call / ReAct 怎么区分

这是 Agent 热门面试高频题。一定要用工程语言答，不要只背定义。

Concept Defense

四者关系

Workflow

流程预先固定，节点怎么走基本由代码决定。适合稳定流水线，比如固定 CI、固定审批、固定报表生成。

Function Call

模型根据语义选择某个函数并填参数，本质是工具调用接口。它解决“怎么让模型调用外部能力”。

ReAct

一种 Agent 推理模式：模型在推理、行动、观察之间循环，用工具观察结果继续推理。它更强调过程中的思考与行动交替。

Agent

Agent 是更完整的系统形态：目标、状态、记忆、规划、工具、观察、评估、纠错、终止条件。Function Call 和 ReAct 都可以是 Agent 的组成部分。

推荐回答：Function Call 解决工具调用接口，ReAct 解决推理和行动交替，Workflow 解决固定流程编排，Agent 则是在目标驱动下动态规划、调用工具、观察结果并迭代的系统。我的项目里固定流程部分会用 Workflow，路径不确定、需要动态选择工具的部分才用 Agent。

Tool / MCP / Skill：工具不是越多越好

严格面试官会追问：工具列表怎么组织？召回后会不会变更工具列表？有什么风险？

Tool Engineering

工具体系怎么讲

为什么要工具召回

工具太多会增加模型选择成本和误调用概率。先根据任务意图召回候选工具，只暴露当前任务需要的工具子集，可以减少干扰、降低 token 成本，也让权限边界更清晰。

召回后变更 tool list 的风险

风险是模型前后上下文不一致、同名工具语义冲突、参数 schema 变化、工具消失导致计划失效。所以工具列表变化要有版本、命名空间和失败兜底。

MCP / Skill 怎么定位

MCP 更像工具接入协议或上下文桥梁，Skill 更像某类高频能力的封装。面试里不用纠结名字，重点说清：输入 schema、权限、执行结果、错误处理和审计。

工具调用失败怎么办

参数校验失败就让模型修正参数；工具执行失败就返回结构化错误；连续失败就停止，输出原因和人工接管建议。写操作需要确认，不让模型无限重试。

工具定义要包含：
- name / namespace：避免同名冲突
- description：模型何时该用
- input_schema：参数类型和约束
- output_schema：返回结构
- permission：只读 / 写入 / 高危
- timeout / retry：超时和重试策略
- error_code：失败原因可被 Agent 理解
- audit_log：记录调用人、参数、结果

追问：为什么不能把所有工具都塞给模型？

工具太多会让模型误选工具，也会放大权限风险。工程上更合理的是按任务意图、权限和上下文做工具召回，只给最小可用工具集。

追问：工具 schema 设计不好会怎样？

模型会填错参数、漏填关键字段，或者把自然语言塞进结构化字段。schema 要尽量明确、枚举化、带示例，并且工具侧必须二次校验。

RAG 与 Memory：上下文不是越多越好

Agent 项目很容易被问：为什么需要 RAG？记忆怎么做？冲突怎么办？

Context & Memory

RAG 和 Memory 的分工

RAG 解决什么

解决项目知识上下文缺失。比如接口文档、Wiki、历史方案、代码片段、故障复盘。它偏事实和知识。

Memory 解决什么

解决长期偏好、历史决策、用户习惯、项目约定。它偏个性化和连续性。

为什么不直接塞全部上下文

上下文窗口有限，塞太多会增加成本、噪声和注意力稀释。RAG 要召回少量高相关内容，并保留来源、时间和置信度。

冲突怎么处理

优先级通常是：当前用户明确指令 > 当前任务规格 > 最新项目文档 > 历史记忆 > 模型常识。冲突时不能悄悄合并，要暴露出来让人确认。

推荐回答：RAG 不等于“搜出来塞进去”，而是上下文工程。要关注 chunk 质量、召回准确率、元信息、去重、时间有效性和引用来源。Memory 也不是无脑记住一切，而是要区分稳定偏好、历史决策和临时上下文。

SubAgent：不是多开几个模型，而是角色化降低复杂度

严格面试官会问：多 Agent 会不会更乱？怎么防止互相甩锅？

SubAgent

我的理解

为什么要 SubAgent

复杂研发任务包含需求、设计、编码、测试、Review 多种认知任务。一个 Agent 一次性做所有事情，容易遗漏。SubAgent 把目标拆窄，产物更稳定。

SubAgent 怎么分

  • 需求分析 Agent：输出澄清问题和验收标准。
  • 方案设计 Agent：输出架构和接口变更。
  • 编码 Agent：输出 diff 和说明。
  • 测试 Agent：输出用例和测试数据。
  • Review Agent：输出风险清单。

多 Agent 风险

上下文不一致、重复劳动、输出冲突、成本上升、链路变长。需要由 Orchestrator 汇总，定义产物格式和冲突解决规则。

怎么收口

每个 SubAgent 产物都要进入统一的任务状态表，最后由主 Agent 或人工做一致性检查，避免多个 Agent 各说各话。

推荐回答：我不会把 SubAgent 当成“模型越多越智能”。它的价值是角色边界清晰、输出可审查。多 Agent 的核心难点反而是调度、一致性和成本控制。

评估与观测：没有 Eval 的 Agent 不能上线

面试官会问你如何证明它有效，以及出了问题怎么定位。

Eval & Observability

指标怎么设计

效果指标

  • 任务完成率
  • 需求到转测耗时
  • 人工修改比例
  • Review 问题数
  • 测试通过率
  • 缺陷逃逸率

Agent 指标

  • 工具选择准确率
  • 工具调用成功率
  • 平均步骤数
  • 重试次数
  • 人工接管率
  • 幻觉/无依据输出比例

工程指标

  • 端到端延迟
  • Token / 成本
  • 上下文召回命中率
  • 失败错误码分布
  • 工具超时率
  • 安全拦截次数
每次运行建议记录 trace：
- user_goal：用户目标
- retrieved_context：召回了哪些文档
- plan：Agent 计划
- tool_calls：调用了哪些工具和参数
- observations：工具返回和错误
- output：最终代码/文档/测试
- human_review：人工修改点
- metrics：耗时、成本、成功/失败原因

追问：生成行数是不是指标？

不是核心指标。生成 3k 行只能说明有真实产物，更关键的是这些代码是否通过测试、是否减少返工、是否提前转测、是否降低 Review 成本。

追问：怎么做回归评估？

可以维护典型需求集和故障集，固定输入跑 Agent，看是否能召回正确上下文、生成合理方案、通过测试，并比较不同版本 Agent 的表现。

迁移模式：可迁移的是 Agent 工程闭环，不是某个 Prompt

回答迁移能力时，要说清什么场景适合 Agent，什么场景不适合。

Migration

适合迁移的场景

订单 / 业务研发

需求澄清、状态机设计、接口草案、幂等方案、代码 diff、单测和转测说明。

运维自动化

自然语言意图、参数校验、MCP 工具执行、日志回传、失败诊断、人工确认。

SQL / 性能排障

日志收集、SQL 解析、EXPLAIN 解读、优化建议、回归验证清单。

知识库 / 新人助手

文档召回、方案总结、历史决策解释、项目 FAQ、故障复盘。

测试生成

根据规格生成单测、边界用例、Mock 数据、回归用例和测试报告。

代码 Review

检查幂等、状态机、异常处理、缓存一致性、SQL 风险、日志和监控。

适合 Agent 的场景：目标明确，但路径需要动态判断；有工具可调用；有可验证结果；有高频重复价值。不适合的场景：规则固定到可以写死流程、结果无法验证、风险极高且没有人工确认。

当前瓶颈：要敢讲不足，才像真的做过

严格面试官更相信你讲得出边界，而不是把 Agent 吹成万能。

Bottlenecks

我会讲 8 个瓶颈

1. 上下文质量

文档过期、chunk 切分差、召回噪声高，会让 Agent 基于错误上下文决策。

2. 工具稳定性

工具 schema 不清、超时、权限不足、错误码不可理解，会导致 Agent 无法恢复。

3. 多步误差累积

Agent loop 越长，前面一步的小错误越可能放大到最终结果。

4. 多文件一致性

AI 写局部代码较强，但跨接口、配置、测试、文档的一致性仍需人工审查。

5. Eval 难做

研发任务开放性强，不能只用对错评估，需要任务集、人工标注、测试和线上指标结合。

6. 成本和延迟

多轮规划、RAG、工具调用、SubAgent 都会增加成本和等待时间，需要按场景取舍。

7. 安全与权限

写文件、执行脚本、热部署等工具必须有沙箱、白名单、审批和审计。

8. 人机责任边界

AI 可以生成候选方案，但架构取舍、上线风险、数据安全和最终合并必须由工程师负责。

下一步方向：把“能跑通”推进到“可复制、可度量、可治理”。重点做工具稳定性、测试自动化、trace 观测、任务集评估、权限边界和人工审核流程。

90 Seconds

Agent 项目 90 秒版本

我近半年参与了 AI 辅助研发与智能交付探索。我不会把它理解成简单的 AI 写代码，而是把它做成一个研发场景下的 Agent 化闭环：用户输入需求后，系统先做需求理解和澄清，再通过 RAG 和历史记忆补项目上下文，然后由 Agent 做任务规划，按需调用 Skill、MCP、代码检索、脚本执行等工具，读取观察结果后继续修正，最后输出代码 diff、测试用例、Review 清单和转测说明。这个过程中我实践过 Skill、MCP、SubAgent、RAG、SDD/TDD 等方式，在试点中完成从 0 到 1 的穿刺验证，累计生成约 3k 行代码，并提前 2 天转测。我沉淀出的模式是：先规格后代码，中间产物可审查，复杂任务角色化，高频动作工具化，人负责判断，AI 负责提高吞吐。现在的瓶颈主要在上下文质量、工具稳定性、多步误差累积、评估体系、安全权限和人机责任边界上。所以我认为 Agent 工程化的关键不是让模型更会聊天，而是把上下文、工具、状态、评估和人工审核都纳入可治理的工程流程。

30 Seconds

Agent 项目 30 秒版本

我做的 AI 辅助研发探索，本质是研发场景的 Agent 工程化。它不是一次性生成代码，而是需求理解、RAG 补上下文、任务规划、工具调用、代码生成、测试验证和人工 Review 的闭环。试点里累计生成约 3k 行代码并提前 2 天转测。我的核心收获是：Agent 的价值不在于替代工程师，而在于把研发流程拆成可审查、可调用工具、可验证结果的多个节点，提升吞吐，同时保留人工判断和风险控制。
