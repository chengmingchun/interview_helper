Data Agent 与面试 AI Coding：从架构到实战

这页只解决两个方向：一个是“让 AI 安全、准确地分析企业数据”的 Data Agent；另一个是“面试里如何用 AI Coding 展示工程能力”。学完要能画架构、讲风险、做工具选型，并把 AI 当成结对工程师而不是答案生成器。

1. 两条主线先讲清楚

Data Agent 和 AI Coding 都是 Agent，但评估标准不同：Data Agent 追求“数字可信”，AI Coding 追求“代码可运行、可维护、可解释”。

Data Agent

让用户用自然语言问业务数据，系统能理解指标、选择数据源、生成查询、执行 SQL、解释结果、画图，并在不确定时追问或拒答。

Text-to-SQLSemantic LayerBI AgentData Governance

AI Coding 面试

在面试或现场 coding 中使用 AI 辅助拆题、写代码、补测试、排错。核心不是“让 AI 代写”，而是展示你能控制需求、验证结果、审查风险。

CodexClaude CodeCursorCopilot
一句话总结：Data Agent 是“自然语言到可信数据分析链路”；AI Coding 是“自然语言到可验证代码变更链路”。两者都要有计划、工具调用、校验和人类审查。

2. Data Agent 是什么

Data Agent 不是简单 Text-to-SQL。Text-to-SQL 只解决“写查询”，Data Agent 还要解决“问得对、查得准、算得对、解释得清、权限安全”。

理解问题识别指标、维度、时间、过滤条件、业务实体。
找数据选择表、视图、指标层、数据集或知识文档。
生成计划决定 SQL、Python、图表、RAG 或追问。
执行工具只读查询、沙箱执行、限制扫描量和超时。
校验结果语法、执行、口径、异常值、行数和权限校验。
解释输出结论、图表、引用 SQL、口径说明、下一步建议。

典型需求

  • “上个月华东地区 GMV 为什么下降？”要拆成指标口径、时间范围、地区维度、同比环比、分渠道/品类归因。
  • “帮我看最近留存异常。”要先定位指标、拉取趋势、检测异常点，再做分群下钻。
  • “生成一个销售日报。”要调用固定指标、生成图表、输出摘要，并保证权限与口径一致。

3. Data Agent 生产架构

用户问题
  -> Intent Router: 数据查询 / 口径解释 / 图表生成 / 异常分析 / 无法回答
  -> Context Builder: 用户权限 + 语义层 + 表结构 + 样例查询 + 指标文档
  -> Planner: 拆任务、选择 SQL / Python / BI / RAG 工具
  -> SQL Generator: 生成只读 SQL，带字段解释和假设
  -> Guardrails: 权限、黑名单、扫描量、超时、敏感字段、SQL 注入
  -> Executor: 查询数据库或沙箱
  -> Verifier: 语法、执行结果、行数、异常值、指标口径、二次查询
  -> Presenter: 结论、图表、SQL、数据来源、置信度、追问建议

Agent 层

负责意图识别、规划、工具选择、循环校验。可用 LangGraph、OpenAI Agents SDK、LlamaIndex Workflows、CrewAI、Mastra 等。

Data Context 层

包括 schema、字段注释、指标定义、血缘、样例 SQL、数据质量规则、权限标签。没有这层，模型只能猜。

Execution 层

执行 SQL/Python/BI 查询，必须只读、限时、限资源、可审计。高风险操作走人工确认。

关键安全边界

风险表现工程处理
越权访问用户问到了无权限表或字段检索 schema 前按 ACL 过滤；SQL 执行层二次校验
错误指标口径把订单数当用户数，把 GMV 当收入语义层、指标字典、query rewrite 中注入口径
SQL 高成本全表扫、无分区、笛卡尔积SQL lint、成本估算、limit、超时、只读副本
静默错误SQL 能跑但数字错执行结果 sanity check、对照查询、异常检测、人工验收集
Prompt injection文档或字段注释诱导模型忽略规则把数据说明当非可信输入，系统指令和工具策略隔离

4. Text-to-SQL 技术点

Text-to-SQL 的难点不在“把中文翻成 SQL”，而在 schema linking、业务口径、join 路径、时间逻辑和结果校验。

生成 SQL 前要给模型什么

  • 相关表结构：表名、字段、类型、注释、主外键。
  • 业务术语：GMV、收入、活跃、留存、退款的定义。
  • 样例 SQL：高质量 few-shot 比泛泛描述更有用。
  • 数据分布：枚举值、时间分区、空值、单位。
  • 权限上下文：哪些库表字段可用。

SQL 生成后要校验什么

  • 语法是否通过 parser/linter。
  • 是否只读，禁止 DDL/DML。
  • 是否命中分区和 limit。
  • join key 是否正确，聚合粒度是否正确。
  • 结果是否为空、极端、重复、违反口径。

Text-to-SQL 常用增强

增强核心思路适合场景
Schema Linking先召回相关表和字段，再生成 SQL库表很多，直接塞全量 schema 不现实
Few-shot SQL给相似业务问题和正确 SQL团队已有分析模板和口径沉淀
SQL Repair执行失败后把错误信息交给模型修复语法、字段名、方言差异问题
Self-check让模型解释 SQL 意图，再检查是否符合问题防止 join 和聚合粒度错误
Execution-guided Decoding生成候选 SQL，执行/验证后选择准确率要求高但可接受更高延迟
Semantic Evidence自动生成或检索字段/值的业务证据业务术语和库表命名差距大
 
生成 SQL 的好提示词骨架：
1. 你只能生成 SELECT 查询。
2. 优先使用给定语义层指标，不要自造口径。
3. 必须说明使用了哪些表、字段、过滤条件和时间范围。
4. 如果问题缺少必要条件，先追问，不要猜。
5. 输出 JSON: { "thought_summary": "...", "sql": "...", "assumptions": [] }

5. 语义层与指标口径

生产 Data Agent 最重要的分水岭：让 LLM 直接写任意 SQL，还是让它通过语义层查询确定性指标。

直接 Text-to-SQL

  • 优点：灵活，能探索任意表和临时问题。
  • 缺点：容易错口径、错 join、错聚合粒度。
  • 适合：数据分析师辅助、低风险探索、内部实验。

语义层优先

  • 优点：指标、维度、实体关系确定，结果更可信。
  • 缺点：覆盖范围取决于建模，没建的指标回答不了。
  • 适合：经营看板、核心指标、面向业务用户。

面试里的正确观点

不是“语义层 vs Text-to-SQL 二选一”，而是分层：核心指标走语义层，探索分析走 Text-to-SQL，复杂归因走 Agent 规划多步查询。语义层回答不了时，系统应该明确说覆盖不足，而不是返回一个看起来正确的错数。

记忆句：Data Agent 的可信度来自“业务语义确定性 + SQL 执行可验证 + 结果解释可追溯”，不是来自模型口才。

7. 面试 AI Coding 的核心打法

面试官看的是你的工程判断：你如何拆需求、约束 AI、验证代码、解释取舍、处理错误。不要把 AI 当黑盒答案机。

复述需求确认输入、输出、边界、复杂度、禁用能力。
先给方案自己讲算法/架构，再让 AI 辅助补代码。
小步生成一次只让 AI 做函数、测试或修 bug。
运行验证跑单测、样例、边界、复杂度检查。
审查修正读 diff，找越界、并发、异常、可维护性问题。
总结表达解释最终方案、复杂度、tradeoff 和 AI 使用方式。

AI Coding 面试禁忌

  • 一上来直接粘题给 AI，不先分析需求和边界。
  • AI 给了代码就提交，不读、不测、不解释。
  • 把复杂问题一次性丢给 AI，导致不可控的大段修改。
  • 隐藏使用 AI 的过程，无法说明每个决策为什么正确。
  • 忽略面试约束：有些公司允许 AI，有些只允许补全，有些完全不允许。

8. 编码智能体工作流

2026 年 AI Coding 已经从 autocomplete 走向 coding agent：能读仓库、制定计划、改代码、跑测试、提交 PR。但你仍然要做 owner。

面试/实战工作流：
1. Clarify: 我先问清楚边界和约束。
2. Plan: 我写出方案和数据结构，不让 AI 先发散。
3. Prompt: 让 AI 只实现一个清晰函数或测试。
4. Verify: 本地运行样例、单测、边界和性能检查。
5. Review: 我读 diff，解释风险和改动原因。
6. Iterate: 失败时把错误日志交给 AI，但我决定修复方向。

10. 面试回答模板

Q1：请你设计一个企业 Data Agent。
我会分成数据上下文层、Agent 编排层、工具执行层和评估观测层。用户提问后先做意图识别和权限校验，再从语义层、schema、字段注释和样例 SQL 中构造上下文。Agent 规划是否需要 Text-to-SQL、指标查询、Python 分析或 RAG 解释。SQL 生成后经过只读校验、权限过滤、成本估算和执行验证，再输出结论、图表、SQL、数据来源和置信度。线上记录 trace，并用黄金问答集持续回归。
Q2：Data Agent 最大风险是什么？
最大风险是静默返回错误数字。代码错了可能报错，数据错了可能看起来很合理。所以要把核心指标放进语义层，SQL 执行前做安全和成本校验，执行后做结果 sanity check，并对关键场景建立人工标注黄金集。
Q3：Text-to-SQL 为什么生产困难？
因为真实库表命名复杂、业务口径隐含、join 路径多、时间逻辑细、权限严格，而且 SQL 能跑不代表数字对。生产系统需要 schema linking、语义层、样例 SQL、执行校验、SQL repair 和人工评估闭环。
Q4：AI Coding 面试时你怎么用 AI？
我会先自己复述需求和边界，再给出解法和复杂度。AI 只用于局部实现、生成测试、检查反例或根据错误日志辅助排查。每次生成后我都会读代码、跑测试、解释 diff 和风险。这样能展示我在控制工程过程，而不是把答案外包给 AI。
Q5：AI 生成代码不对怎么办？
先用测试和日志定位错误，再判断是需求理解错、边界漏了、状态管理错了还是复杂度不满足。然后把最小错误信息反馈给 AI，让它修一个局部问题。我不会让 AI 大范围重写，除非我已经确认原方案方向错了。
Q6：如何评价一个 AI Coding Agent？
看它是否能理解仓库、制定可执行计划、做小步修改、运行测试、解释 diff、遵守权限边界，并在失败时利用日志迭代。企业场景还要看安全、审计、私有代码保护、成本和团队工作流集成。
